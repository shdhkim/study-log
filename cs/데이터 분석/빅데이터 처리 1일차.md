# 빅데이터 처리 흐름 총정리

(HDFS, S3, Spark, Hadoop, Kubernetes, Driver, Executor, Shuffle 포함)

---

## 1. 전체 흐름

우리가 지금까지 공부한 내용을 한 줄로 줄이면 아래와 같다.

```text
로그 생성
→ 로그 수집
→ 저장소(HDFS 또는 S3)에 저장
→ Spark가 읽어서 처리
→ 결과를 DB나 파일에 저장
→ 사람이 조회
```

즉 핵심은 이것이다.

```text
원본 로그를 모은다
→ 저장한다
→ 여러 서버가 나눠서 계산한다
→ 최종 결과를 저장한다
```

---

## 2. 대표 예시

예를 들어 쇼핑몰 서버가 아래 같은 로그를 남긴다고 하자.

```text
2026-03-21 10:00:01 GET /login 200 user1
2026-03-21 10:00:03 GET /main 200 user1
2026-03-21 10:00:05 POST /order 500 user2
2026-03-21 10:00:07 GET /product/100 200 user3
2026-03-21 10:00:08 POST /order 500 user4
2026-03-21 10:00:09 GET /main 200 user2
```

각 줄은 "로그 1건"이다.

각 항목은 대충 이렇게 이해하면 된다.

- 날짜/시간
- HTTP Method
- URL
- 상태 코드
- 사용자

여기서 예를 들어 우리가 알고 싶은 게 다음과 같다고 하자.

1. 어떤 URL에서 에러가 많이 발생했는가
2. 상태 코드 500은 몇 건인가
3. 사용자별 요청 수는 몇 건인가

---

## 3. 로그는 어디서 생기나

로그는 보통 아래 같은 곳에서 생성된다.

- 웹 서버
- 애플리케이션 서버
- 배치 서버
- DB 연동 프로그램
- 장비/센서
- 모바일 앱 이벤트 수집기

예를 들어 서버 3대가 있다고 하자.

```text
Server A → app-a.log
Server B → app-b.log
Server C → app-c.log
```

각 서버가 자기 로그 파일을 계속 만든다.

문제는 로그가 흩어져 있다는 점이다.  
분석하려면 결국 한 군데로 모아야 한다.

---

## 4. 로그 수집은 어떻게 하나

수집은 크게 2가지 방식으로 생각하면 된다.

### 4-1. 파일 수집 방식

각 서버에 로그 파일이 생기고, 주기적으로 중앙 저장소로 보낸다.

```text
서버에서 log 파일 생성
→ 수집 스크립트 또는 에이전트가 파일 읽음
→ HDFS 또는 S3에 업로드
```

예를 들어:

```text
/var/log/app-a.log
/var/log/app-b.log
/var/log/app-c.log
```

이 파일들을 5분마다, 1시간마다, 또는 하루마다 업로드한다.

이 방식은 단순하다.  
대신 거의 실시간은 아닐 수 있다.

---

### 4-2. Kafka 같은 메시지 큐 방식

로그가 생길 때마다 메시지 큐로 보낸다.

```text
앱 서버
→ Kafka
→ Consumer
→ HDFS 또는 S3 저장
```

이 방식은 거의 실시간에 가깝다.

예를 들어 웹 서버가 주문 에러를 발생시키면 바로 Kafka로 보낼 수 있다.

```text
POST /order 500 user2
```

그러면 Consumer가 받아서 S3나 HDFS에 적재한다.

---

## 5. 저장은 왜 따로 하냐

많은 사람이 여기서 헷갈린다.

"그냥 DB에 넣고 SQL 돌리면 되는 거 아닌가?"

작은 데이터면 가능하다.  
하지만 로그는 보통 아래 특성이 있다.

- 양이 엄청 많다
- 텍스트 파일 형태가 많다
- 컬럼이 고정되어 있지 않을 수도 있다
- 서버 여러 대에서 동시에 생긴다

그래서 원본 로그는 보통 **저장소**에 먼저 쌓는다.

여기서 대표 저장소가:

- HDFS
- S3

다.

---

## 6. HDFS는 어떤 역할인가

HDFS는 Hadoop Distributed File System이다.

쉽게 말하면:

```text
큰 파일을 여러 서버에 나눠 저장하는 분산 파일 시스템
```

### 6-1. 예시

하루 로그가 1GB라고 하자.

```text
2026-03-21-log.txt (1GB)
```

이 파일을 HDFS에 넣으면 내부적으로 블록으로 쪼갠다.

예를 들어 블록 크기가 128MB라면:

```text
Block1
Block2
Block3
...
Block8
```

그리고 여러 서버에 나눠 저장한다.

```text
Node1 → Block1, Block5
Node2 → Block2, Block6
Node3 → Block3, Block7
Node4 → Block4, Block8
```

즉 파일 하나를 한 서버에 통째로 두는 게 아니라  
조각내서 여러 서버에 저장하는 구조다.

### 6-2. 왜 이렇게 하냐

이유는 2개다.

1. 용량 확장  
   서버 1대로는 한계가 있으니 여러 서버를 묶어서 저장 공간을 늘린다.

2. 병렬 처리  
   나중에 읽을 때 여러 서버가 동시에 읽을 수 있다.

### 6-3. HDFS 구성

보통 아래처럼 이해하면 된다.

- NameNode: 파일 메타데이터 관리
- DataNode: 실제 데이터 저장

예를 들어 NameNode는 이런 정보를 안다.

```text
2026-03-21-log.txt의 Block1은 Node1과 Node2에 있다
```

DataNode는 진짜 파일 조각을 갖고 있다.

## 7. S3는 어떤 역할인가

S3는 Amazon Simple Storage Service다.

쉽게 말하면:

```text
클라우드에 있는 대형 저장 창고
```

그리고 더 정확히 말하면:

```text
객체 스토리지
```

### 7-1. S3는 처리 안 한다

이게 매우 중요하다.

S3는:

- 계산 안 함
- 집계 안 함
- 분석 안 함

오직 저장만 한다.

즉 역할은 아래다.

```text
원본 로그를 쌓아두는 저장소
```

### 7-2. 예시

S3 버킷이 있다고 하자.

```text
s3://shop-log-bucket/
```

그 안에 날짜별로 로그를 저장할 수 있다.

```text
s3://shop-log-bucket/2026/03/21/app-a.log
s3://shop-log-bucket/2026/03/21/app-b.log
s3://shop-log-bucket/2026/03/21/app-c.log
```

### 7-3. 왜 S3를 쓰냐

1. 확장 쉬움  
   서버 직접 관리 안 해도 된다.

2. 운영 편함  
   HDFS처럼 NameNode/DataNode 직접 운영 안 해도 된다.

3. 안정성 높음  
   복제, 내구성, 장애 처리 등을 클라우드가 많이 대신해준다.

### 7-4. HDFS와 S3 차이

핵심만 요약하면 아래다.

| 구분      | HDFS                      | S3                   |
| --------- | ------------------------- | -------------------- |
| 성격      | 분산 파일 시스템          | 객체 스토리지        |
| 위치      | 보통 온프레미스/직접 운영 | 클라우드             |
| 관리      | 직접 운영                 | 클라우드가 관리      |
| 읽기      | 로컬성 활용 가능          | 네트워크로 읽음      |
| 대표 역할 | 내부 대용량 저장          | 클라우드 대용량 저장 |

중요한 점은:

```text
S3는 Hadoop 전체 대체가 아니라
HDFS의 저장 역할을 대체하는 쪽에 가깝다
```

즉:

- HDFS = 저장
- S3 = 저장
- Spark = 처리

이다.

## 8. Hadoop은 정확히 뭐냐

많이 헷갈리는 부분이라 정확히 정리한다.

Hadoop을 크게 나누면 보통 아래로 본다.

1. HDFS  
   데이터 저장

2. YARN  
   자원 관리

3. 예전의 MapReduce  
   분산 처리 방식

요즘은 실제 처리 엔진으로 MapReduce보다 Spark를 더 많이 쓴다.

즉 실무적으로는:

```text
HDFS = 저장
YARN = CPU/메모리 같은 자원 관리
Spark = 실제 계산
```

이렇게 이해하면 된다.

---

## 9. Spark는 정확히 무슨 역할인가

Spark는 **분산 처리 엔진**이다.

쉽게 말하면:

```text
여러 서버가 동시에 계산하게 해주는 시스템
```

예를 들어 우리가 하고 싶은 작업이:

```text
상태 코드 500인 URL별 에러 건수 집계
```

라고 하자.

Spark는 이걸 여러 서버에 나눠서 동시에 계산하게 한다.

즉 역할은 아래다.

- 데이터 읽기
- 필터링
- 그룹핑
- 집계
- 조인
- 결과 저장

중요한 점은:

```text
Spark는 저장소가 아니다
Spark는 처리기다
```

## 10. Spark 코드는 대충 어떻게 생기나

아래는 PySpark 예시다.

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import split, col

spark = SparkSession.builder.appName("web-log-analysis").getOrCreate()

df = spark.read.text("s3://shop-log-bucket/2026/03/21/")

parsed = df.withColumn("parts", split(col("value"), " ")) \
    .selectExpr(
        "parts[0] as date",
        "parts[1] as time",
        "parts[2] as method",
        "parts[3] as url",
        "parts[4] as status",
        "parts[5] as user_id"
    )

error_df = parsed.filter(col("status") == "500")

result = error_df.groupBy("url").count()

result.show()
```

이 코드의 의미는 아래와 같다.

1. 로그 파일 읽기
2. 문자열 쪼개기
3. status가 500인 것만 남기기
4. URL별로 묶기
5. 개수 세기

결과는 예를 들어 이렇게 나올 수 있다.

```text
/order      2
/payment    5
/login      1
```

## 11. spark-submit은 무슨 역할인가

많이 헷갈리는 포인트다.

`spark-submit`은 자동 분석 버튼이 아니다.

정확히는:

```text
내가 작성한 Spark 프로그램을 클러스터에 실행시키는 명령어
```

예:

```bash
spark-submit job.py
```

이 의미는:

```text
job.py 안에 작성한 코드를
클러스터 전체에서 실행해라
```

이지,

```text
알아서 데이터를 이해해서 분석해라
```

가 아니다.

즉:

- 로직은 내가 작성
- 실행은 Spark가 분산으로 해줌

이다.

## 12. Driver와 Executor는 뭐냐

이 부분이 Spark 이해의 핵심이다.

### 12-1. Driver

Driver는 중앙 컨트롤러 느낌이다.

역할:

- 코드 실행 계획 수립
- 작업 분할
- Executor에 작업 지시
- 결과 수신

### 12-2. Executor

Executor는 실제 작업자다.

역할:

- 데이터 읽기
- 필터링
- 그룹핑
- 집계
- 중간 결과 생성

즉 한 줄 요약하면:

```text
Driver = 지시
Executor = 실행
```

---

## 13. 진짜 실행은 어디서 되나

예를 들어 서버가 4대 있다고 하자.

```text
Server1
Server2
Server3
Server4
```

로그는 HDFS나 S3에 저장되어 있다.

Spark job을 실행하면:

- Driver 1개가 생김
- Executor 여러 개가 생김

예를 들어:

```text
Driver → Server1
Executor1 → Server2
Executor2 → Server3
Executor3 → Server4
```

또는 Kubernetes면 Pod로 실행된다.

즉 실제 계산은 여러 서버에서 동시에 일어난다.

중요한 점:

```text
서버는 물리/가상 머신이고
Python은 그 위에서 실행되는 코드다
```

즉 "서버 = Python 서버"가 아니다.

---

## 14. Kubernetes에서는 어떻게 되나

Spark를 Kubernetes 위에서 돌릴 수도 있다.

이 경우 구조는 대략 아래다.

```text
Driver Pod 1개
Executor Pod 여러 개
```

흐름은 이렇다.

1. `spark-submit` 실행
2. Kubernetes가 Driver Pod 생성
3. Driver가 Executor Pod 여러 개 요청
4. Kubernetes가 여러 노드에 Executor Pod 생성
5. Executor Pod들이 데이터 처리
6. 결과를 Driver로 전달
7. 작업 끝나면 Pod 종료

즉 Kubernetes의 역할은:

```text
Spark를 실행할 Pod를 띄워주는 플랫폼
```

이다.

실제 계산은 Executor Pod가 한다.

---

## 15. Hadoop/YARN에서는 어떻게 되나

Spark를 Hadoop 위에서 돌리면 보통 YARN을 사용한다.

흐름은 이렇다.

1. `spark-submit --master yarn job.py`
2. Driver가 YARN에 자원 요청
3. YARN이 적절한 서버를 선택
4. 그 서버들에 Executor 생성
5. Executor가 데이터 처리

즉 YARN의 역할은:

```text
클러스터 자원(CPU, 메모리)을 관리하고 Executor를 배치
```

이다.

---

## 16. 데이터는 어떻게 나눠서 처리하나

여기서 진짜 중요한 개념이 나온다.

예를 들어 아래 로그가 있다고 하자.

```text
2026-03-21 10:00:01 GET /login 200 user1
2026-03-21 10:00:03 GET /main 200 user1
2026-03-21 10:00:05 POST /order 500 user2
2026-03-21 10:00:07 GET /product/100 200 user3
2026-03-21 10:00:08 POST /order 500 user4
2026-03-21 10:00:09 GET /main 200 user2
```

이 데이터가 3개 서버에 나뉘어 있다고 하자.

```text
Executor1 → 앞부분 일부
Executor2 → 중간 일부
Executor3 → 뒷부분 일부
```

각 Executor는 자기 데이터만 읽는다.

예를 들어 상태코드 500만 필터하면:

- Executor1 → 없음
- Executor2 → `/order 500`
- Executor3 → `/order 500`

그 다음 URL별 count를 해야 하니 같은 URL끼리 모아야 한다.

## 17. Shuffle은 뭐냐

Shuffle은:

```text
같은 key끼리 데이터를 모으기 위해
Executor들 사이에서 데이터를 이동시키는 과정
```

이다.

예를 들어 Executor들이 부분 결과를 아래처럼 만들었다고 하자.

```text
Executor1 → /login = 1
Executor2 → /order = 1
Executor3 → /order = 1
```

최종적으로 `/order = 2`를 만들려면  
`/order` 데이터가 한 군데로 모여야 한다.

이때 데이터가 네트워크를 타고 이동한다.

이게 Shuffle이다.

즉:

```text
각자 계산
→ 같은 key끼리 이동
→ 최종 집계
```

이다.

---

## 18. 최종 집계는 누가 하나

중요한 포인트다.

정답은:

```text
Driver가 집계하는 게 아니라
Executor가 집계를 한다
```

이다.

흐름은 보통 이렇다.

1. Executor들이 부분 계산
2. Shuffle로 같은 key 이동
3. 특정 Executor가 최종 집계
4. Driver는 최종 결과만 받음

즉:

- 계산은 Executor
- 제어는 Driver

이다.

---

## 19. 예시로 최종 집계까지 다시 보기

로그가 있다고 하자.

```text
GET /login 200 user1
POST /order 500 user2
POST /order 500 user4
GET /main 200 user2
```

서버별로 나뉘면:

```text
Executor1 → GET /login 200 user1
Executor2 → POST /order 500 user2
Executor3 → POST /order 500 user4, GET /main 200 user2
```

500만 남기면:

```text
Executor1 → 없음
Executor2 → /order
Executor3 → /order
```

부분 집계:

```text
Executor2 → /order = 1
Executor3 → /order = 1
```

Shuffle 후:

```text
/order 관련 결과가 한 Executor로 모임
```

최종 집계:

```text
/order = 2
```

Driver는 이 최종 결과를 받는다.

## 20. HDFS에서 읽는다는 말은 정확히 무슨 뜻인가

많이 하는 오해가 있다.

```text
HDFS 데이터를 한 서버로 몽땅 가져와서 처리한다
```

라고 생각하기 쉽다.

실제로는 그렇지 않다.

HDFS는 데이터가 여러 노드에 나뉘어 있으므로  
가능하면 데이터가 있는 곳에서 Executor가 읽는다.

즉 개념적으로는:

```text
데이터를 옮기기보다
계산을 데이터 가까이 보낸다
```

라고 이해하면 된다.

이걸 data locality라고 한다.

반면 S3는 보통 네트워크를 통해 읽는다.

즉:

- HDFS: 로컬성 활용 가능
- S3: 네트워크 기반 읽기

차이가 있다.

---

## 21. 그럼 HDFS와 S3 중 뭘 쓰냐

환경에 따라 다르다.

### 온프레미스/사내 데이터센터

보통 아래를 많이 생각한다.

- HDFS
- 또는 MinIO 같은 S3 호환 스토리지

### 클라우드

보통 아래가 많다.

- S3 + Spark
- Databricks, EMR, Glue 같은 서비스와 함께

즉 요즘 많이 보는 구조는:

```text
S3 = 저장
Spark = 처리
```

이다.

전통적인 구조는:

```text
HDFS = 저장
YARN = 자원관리
Spark = 처리
```

이다.

## 22. 전체 아키텍처 예시 1: HDFS + Spark

```text
웹 서버들
→ 로그 파일 생성
→ 수집기
→ HDFS 저장
→ Spark 처리
→ 결과를 Oracle 저장
→ 대시보드 조회
```

조금 더 자세히 쓰면:

```text
Server A/B/C에서 로그 생성
→ Filebeat 또는 배치 스크립트가 수집
→ HDFS에 날짜별로 적재
→ Spark Job이 HDFS 읽음
→ URL별 에러 건수 집계
→ Oracle의 DAILY_ERROR_STAT 테이블에 저장
→ 운영 화면에서 조회
```

## 23. 전체 아키텍처 예시 2: S3 + Spark + Kubernetes

```text
웹 서버들
→ Kafka로 로그 전송
→ Consumer가 S3 저장
→ Spark on Kubernetes가 S3 읽음
→ 집계 결과를 PostgreSQL 저장

```

## 24. 실제 PySpark 예시

아래는 웹 로그에서 상태코드 500 URL별 건수를 세는 예시다.

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import split, col

spark = SparkSession.builder \
    .appName("web-log-error-analysis") \
    .getOrCreate()

# 1. 원본 로그 읽기
df = spark.read.text("s3://shop-log-bucket/2026/03/21/")

# 2. 문자열 분리
parsed = df.withColumn("parts", split(col("value"), " ")) \
    .select(
        col("parts")[0].alias("date"),
        col("parts")[1].alias("time"),
        col("parts")[2].alias("method"),
        col("parts")[3].alias("url"),
        col("parts")[4].alias("status"),
        col("parts")[5].alias("user_id")
    )

# 3. 500 에러만 추출
error_df = parsed.filter(col("status") == "500")

# 4. URL별 에러 건수 집계
result = error_df.groupBy("url").count()

# 5. 결과 확인
result.show()

# 6. 결과 저장
result.write \
    .format("jdbc") \
    .option("url", "jdbc:postgresql://db-host:5432/logdb") \
    .option("dbtable", "url_error_stat") \
    .option("user", "appuser") \
    .option("password", "password") \
    .save()
```

### 이 코드가 실제로 하는 일

1. S3에서 원본 로그 읽기
2. 각 줄을 컬럼처럼 파싱
3. status = 500만 남김
4. URL별로 count
5. 결과를 DB 저장

## 25. Spark는 자동 분석 도구가 아니다

이 부분 다시 한 번 강조한다.

Spark는 아래가 아니다.

```text
데이터 넣으면 알아서 의미를 파악해서 분석해주는 도구
```

Spark는 아래다.

```text
내가 작성한 코드를
여러 서버에서 병렬 실행해주는 엔진
```

즉:

- 분석 로직은 내가 설계
- 분산 실행은 Spark가 담당

이다.

## 26. 개념 관계 정리

헷갈리기 쉬운 개념을 한 번에 정리하면 아래와 같다.

| 구성       | 역할                                  |
| ---------- | ------------------------------------- |
| 로그       | 원본 데이터                           |
| Kafka      | 수집/전달                             |
| HDFS       | 분산 저장                             |
| S3         | 객체 저장                             |
| Hadoop     | HDFS + YARN 같은 저장/자원관리 생태계 |
| YARN       | 자원 관리                             |
| Spark      | 분산 처리 엔진                        |
| Driver     | 작업 지시/제어                        |
| Executor   | 실제 계산                             |
| Kubernetes | Pod 실행 플랫폼                       |
| DB         | 최종 결과 저장/조회                   |

## 27. 핵심 요약

### 핵심 1

원본 로그는 먼저 저장소에 쌓는다.

- HDFS
- S3

### 핵심 2

실제 계산은 Spark가 한다.

- filter
- groupBy
- join
- count

### 핵심 3

Spark는 여러 서버에 Executor를 띄워서 나눠 계산한다.

### 핵심 4

같은 key끼리 모아야 할 때 Shuffle이 발생한다.

### 핵심 5

최종 집계도 Driver가 아니라 Executor가 한다.

### 핵심 6

Driver는 전체 작업을 조율하고 결과를 받는다.

## 28. 내용 정리

쇼핑몰 주문 장애 로그를 종이 문서라고 생각해보자.

- 창고 = S3/HDFS
- 팀장 = Driver
- 직원들 = Executor
- 회사 운영실 = YARN/Kubernetes

흐름은 아래다.

1. 종이 문서(로그)가 창고에 쌓인다
2. 팀장이 직원 여러 명을 배치한다
3. 직원들이 각자 일부 문서를 읽는다
4. "주문 실패" 같은 것만 추린다
5. 같은 종류끼리 모아서 합친다
6. 최종 집계를 만든다
7. 팀장이 결과 보고만 받는다

이게 Spark + HDFS/S3 구조의 핵심이다.

## 30. 단어 요약

### HDFS

여러 서버에 파일을 쪼개 저장하는 분산 파일 시스템

### S3

클라우드 객체 저장소, 저장만 담당

### Hadoop

HDFS/YARN 중심의 저장 + 자원관리 생태계

### Spark

실제 계산을 담당하는 분산 처리 엔진

### Driver

작업 계획 세우고 지시하는 중앙 제어자

### Executor

실제 데이터를 읽고 계산하는 작업자

### Shuffle

같은 key끼리 모으기 위해 Executor 사이에서 데이터 이동하는 과정

### Kubernetes

Driver Pod, Executor Pod를 띄워 Spark 작업을 실행시키는 플랫폼
