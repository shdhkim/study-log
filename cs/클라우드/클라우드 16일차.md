# Spring Cloud Eureka와 LoadBalancer 동작 정리

## 1. 전체 구조

Spring Cloud에서 서비스 간 통신 구조는 다음과 같이 동작한다.

    Client
       ↓
    API Gateway
       ↓
    order-service
       ↓
    Spring Cloud LoadBalancer
       ↓
    Eureka Server
       ↓
    user-service instance

각 구성 요소 역할

  구성요소                    역할
  --------------------------- --------------------
  Eureka Server               서비스 목록 관리
  Spring Cloud LoadBalancer   서버 선택
  Feign / RestTemplate        HTTP 요청 수행
  Microservice                실제 비즈니스 로직

------------------------------------------------------------------------

# 2. Eureka Server 역할

Eureka 서버는 **서비스 위치를 저장하는 서버**이다.

즉 어떤 서비스가 어디에서 실행 중인지 관리한다.

예를 들어 user-service가 3개 떠 있으면

    USER-SERVICE
     ├ 10.0.1.5:8081
     ├ 10.0.1.6:8081
     └ 10.0.1.7:8081

이 정보를 Eureka가 저장한다.

즉

    서비스 이름 → 실제 서버 목록

을 관리하는 **서비스 레지스트리(Service Registry)** 이다.

------------------------------------------------------------------------

# 3. 서비스 등록 과정

서비스가 시작되면 Eureka 서버에 등록된다.

예

    user-service 시작

Eureka 등록 요청

    POST /eureka/apps/USER-SERVICE

등록 결과

    USER-SERVICE
     ├ 10.0.1.5

------------------------------------------------------------------------

# 4. Heartbeat (서버 생존 확인)

서비스는 주기적으로 다음 신호를 보낸다.

    나 살아있음

요청

    PUT /eureka/apps/USER-SERVICE/{instanceId}

기본 주기

  설정        기본값
  ----------- --------
  heartbeat   30초

즉

    서비스 → Eureka

로 계속 상태를 보고한다.

------------------------------------------------------------------------

# 5. 서버가 죽었을 때

예

    10.0.1.6 서버 다운

그러면

    heartbeat 전송 중단

Eureka는 일정 시간 기다린다.

  설정       기본값
  ---------- --------
  eviction   90초

즉

    90초 동안 heartbeat 없음 → 서버 제거

Eureka registry

제거 전

    USER-SERVICE
     ├ 10.0.1.5
     ├ 10.0.1.6
     └ 10.0.1.7

제거 후

    USER-SERVICE
     ├ 10.0.1.5
     └ 10.0.1.7

------------------------------------------------------------------------

# 6. 로드밸런싱은 누가 하는가

많이 헷갈리는 부분이다.

로드밸런싱은

    Eureka 서버가 하지 않는다

로드밸런싱은

    Spring Cloud LoadBalancer

가 수행한다.

------------------------------------------------------------------------

# 7. 실제 서비스 호출 흐름

예

    order-service → user-service 호출

코드

    http://user-service/users/1

실제 내부 동작

    order-service
       ↓
    Spring Cloud LoadBalancer
       ↓
    Eureka 조회
       ↓
    instance 목록
       ↓
    하나 선택
       ↓
    HTTP 요청

------------------------------------------------------------------------

# 8. instance 선택 방식

대표 알고리즘

  방식          설명
  ------------- ---------------
  Round Robin   순서대로 선택
  Random        랜덤 선택

예

    요청1 → instance1
    요청2 → instance2
    요청3 → instance3
    요청4 → instance1

------------------------------------------------------------------------

# 9. Client-side Load Balancing 개념

Spring Cloud에서는

    요청 보내는 서비스가 직접 서버를 선택

한다.

즉

    order-service

가 로드밸런싱을 수행한다.

그래서 이름이

    Client-side Load Balancing

이다.

------------------------------------------------------------------------

# 10. 전체 정리

각 컴포넌트 역할

  구성요소                    역할
  --------------------------- ------------------
  Eureka Server               서비스 목록 관리
  Service Instance            heartbeat 전송
  Spring Cloud LoadBalancer   서버 선택
  Feign / RestTemplate        HTTP 요청 수행

------------------------------------------------------------------------

# 11. 전체 동작 흐름

Spring Cloud 서비스 호출 흐름

    서비스 시작
       ↓
    Eureka 등록
       ↓
    Heartbeat 전송
       ↓
    Eureka registry 유지
       ↓
    LoadBalancer가 instance 선택
       ↓
    서비스 호출
