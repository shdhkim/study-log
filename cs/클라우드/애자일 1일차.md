# **애자일(Agile)**
소프트웨어 개발 및 프로젝트 관리에서 **유연성(flexibility)**과 **반복적(iterative) 개선**을 중심으로 하는 방법론.  
기존의 **폭포수(Waterfall)** 방식처럼 계획을 처음부터 끝까지 한 번에 세우는 것이 아니라,  
짧은 주기(**Sprint**)로 나누어 점진적으로 개선하며 개발하는 방식.

## **애자일의 장점**
✅ 변화에 유연하게 대응 가능  
✅ 빠른 피드백 & 개선  
✅ 출시 속도 향상  
✅ 팀 간 협업 & 커뮤니케이션 강화  
✅ 고객 만족도 증가  

---

# **애자일 프레임워크**
## **스크럼(Scrum)**
가장 많이 사용되는 애자일 프레임워크.  
- 일정한 주기로 기능을 개발하는 프로젝트에 적합  
- **팀워크**와 **주기적인 개선**에 초점  

## **칸반(Kanban)**
시각적인 작업 흐름 관리 (보드 사용).  
- **지속적인 작업 흐름**을 강조  
- **유연한 작업 관리가 필요할 때 적합**  
- 운영, 고객 지원, 지속적 개선 업무에 활용  

### **📌 스크럼 vs 칸반**
| | **스크럼(Scrum)** | **칸반(Kanban)** |
|---|---|---|
| **방식** | 일정한 주기(Sprint)마다 기능 개발 | 지속적인 요청 처리 |
| **목적** | 기능 개발 및 반복적 개선 | 유연한 작업 관리 |
| **사용 사례** | 소프트웨어 개발, 금융 IT 프로젝트 | 고객 지원, 운영 관리 |

## **XP(Extreme Programming)**
애자일 개발을 강화하는 프로그래밍 기법.

# **애자일 예제**
## **📌 전자 행정 시스템 (폭포수 모델)**
한 정부 기관이 새로운 전자 행정 시스템을 구축하는 프로젝트.  
- 명확한 요구사항과 법적 기준을 따라야 함.

1. **요구사항 분석(Requirements)**  
   - 고객(정부 기관)과 미팅을 진행해 시스템의 기능과 요구사항을 정의.  
2. **설계(Design)**  
   - 데이터베이스 및 시스템 아키텍처 설계.  
3. **구현(Implementation)**  
   - 설계된 내용을 바탕으로 코딩 시작.  
4. **테스트(Testing)**  
   - 전체 시스템 통합 후 QA(품질 보증) 테스트 진행.  
5. **배포(Deployment)**  
   - 시스템을 정식으로 배포.  

## **📌 헬스케어 앱 개발 (애자일 방식)**
한 스타트업이 사용자 맞춤형 헬스케어 앱을 개발.  
- **사용자 피드백을 반영하며 지속적으로 개선**해야 함.  

### **💡 개발 프로세스**
1. **제품 백로그(Backlog) 생성**  
   - 기본 기능 정의: **사용자 로그인, 운동 기록 저장, AI 운동 추천**  
2. **스프린트(Sprint) 진행 (2주 단위)**
   - **첫 번째 스프린트:** 로그인 기능 개발  
   - **두 번째 스프린트:** 운동 기록 저장 기능 추가  
   - **세 번째 스프린트:** AI 운동 추천 기능 추가  
3. **출시 & 지속적인 업데이트**  
   - **MVP(Minimum Viable Product) 출시** 후, 지속적인 개선 진행.

# **스마트 팩토리(Smart Factory)**
AI, IoT, 빅데이터, 클라우드, 로봇, 자동화 기술을 활용해  
**생산 공정을 최적화하는 공장 시스템**.  
기존 공장보다 **자동화, 데이터 분석, 연결성이 강화된 차세대 공장**.  

💡 **비유:**  
- **이미지 = 붕어빵 틀**  
- **컨테이너 = 붕어빵**

# **오토스케일링(Auto Scaling)**
시스템의 **부하(트래픽, CPU 사용률 등)**에 따라  
자동으로 서버의 수를 조절하는 기능.  

## **오토스케일링이 필요한 이유**
1. **갑작스러운 트래픽 증가 대응**  
   - 예제: 쇼핑몰 할인 이벤트로 접속자가 폭발적으로 증가할 경우  
   - 오토스케일링이 없으면 **서버 과부하 → 서비스 장애 발생**  
   - 오토스케일링이 있으면 **자동으로 서버 추가 → 원활한 서비스 제공**  

2. **비용 절감**  
   - 평소에는 트래픽이 적지만 특정 시간(예: 점심시간, 출근 시간)에 급증할 경우  
   - **유휴 서버를 줄여 불필요한 비용 절감 가능**  

3. **서비스 안정성 보장**  
   - **서버 장애 발생 시 자동으로 대체 서버 배포 → 무중단 서비스 유지**

# **백로그(Backlog)**
프로젝트에서 해야 할 **모든 작업의 리스트**.  
- **기능 개발, 버그 수정, 기술 부채 해결, 개선 작업** 등을 포함.  

## **제품 백로그(Product Backlog)**
- 전체 제품(서비스)에 필요한 기능과 개선 사항을 정리한 목록.  
- **제품 책임자(Product Owner, PO)**가 관리.  
- **고객의 요구사항 & 시장 변화에 따라 지속적으로 업데이트됨.**  

### **예제**
| 기능 | 설명 |
|---|---|
| 사용자 로그인 기능 | 이메일 & 비밀번호 로그인 지원 |
| 상품 검색 기능 | 키워드 기반 검색 기능 구현 |
| 장바구니 기능 | 선택한 상품을 저장하고 결제 가능 |
| 추천 상품 시스템 | AI 기반으로 추천 상품 제공 |

## **스프린트 백로그(Sprint Backlog)**
- **제품 백로그에서 특정 스프린트(2~4주 단위)에 수행할 작업을 선정한 목록**.  
- **개발팀이 직접 관리하며, 실행 가능한 단위로 나눔**.  

### **예제**
| 작업 | 예상 시간 |
|---|---|
| 로그인 UI 디자인 | 4시간 |
| 로그인 API 개발 | 6시간 |
| 로그인 기능 테스트 | 2시간 |

# **방법론 테일러링(Tailoring)**
기존 방법론을 그대로 따르는 것이 아니라,  
**프로젝트 상황에 맞게 조정하는 것**.  

- **팀, 프로젝트 크기, 일정, 요구사항 변화 등에 따라 유연하게 방법론 적용**.  
- **소프트웨어 개발뿐만 아니라, 프로젝트 관리(PMBOK)에서도 자주 사용됨**.  

## **테일러링의 필요성**
- 모든 프로젝트가 동일한 환경에서 진행되지 않음.  
- 기존 방법론을 무조건 적용하면 **비효율적일 수 있음**.  
- 따라서, **프로젝트 특성에 맞춰 적절히 변경하는 것이 중요**.  

## **테일러링 적용 사례**
### **프로젝트 개발 방법론**
✅ **애자일(Agile) + 폭포수(Waterfall) 하이브리드 적용**  
   - **초기 분석 & 설계** → 폭포수 방식  
   - **개발 & 테스트** → 애자일 적용  

✅ **스크럼(Scrum) + 칸반(Kanban) 혼합**  
   - **개발팀** → **스프린트(Sprint) 방식 적용**  
   - **운영팀** → **칸반(Kanban) 방식 사용**  

### **📌 금융 IT 프로젝트 테일러링**
- **금융 시스템**은 안정성이 중요 → 처음부터 **상세한 설계 필요 (폭포수 적용)**  
- **그러나 고객 요구가 자주 변경됨** → 일부 기능 개발은 **애자일 방식 적용**
