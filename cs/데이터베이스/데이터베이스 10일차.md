# 오라클(Oracle) 트랜잭션 격리수준 & 이상현상 정리 


> **격리수준은 "읽기 일관성", 락/버전은 "쓰기 충돌" 해결**

------------------------------------------------------------------------

## 1. 오라클의 기본 구조 이해 (MVCC)

오라클은 **MVCC(Multi Version Concurrency Control)** 기반 DB이다.

-   SELECT → 과거 스냅샷(Undo)을 읽음 → 거의 안막힘
-   UPDATE/DELETE → 실제 행을 수정 → 행 배타락(X-lock) 필요 → 서로 대기

  작업     동작
  -------- -----------------------
  SELECT   스냅샷 읽기(락 없음)
  UPDATE   행 락 필요(대기 발생)

예시

``` sql
-- A 트랜잭션
UPDATE account SET balance = 0 WHERE id = 1; -- 아직 COMMIT 안함

-- B 트랜잭션
SELECT balance FROM account WHERE id = 1; -- 대기하지 않고 이전 값 조회
```

**왜?**\
Undo 영역의 이전 버전을 읽기 때문이다.

------------------------------------------------------------------------

## 2. 오라클의 격리수준

오라클에서 실질적으로 의미 있는 격리수준은 2개뿐이다.

1.  READ COMMITTED (기본)
2.  SERIALIZABLE

※ READ UNCOMMITTED 없음\
※ REPEATABLE READ 없음 (MVCC로 일부 효과 포함)

------------------------------------------------------------------------

## 3. 동시성 이상현상 6가지

### 1) Dirty Read (더티 리드)

커밋되지 않은 데이터 읽기

오라클: **발생하지 않음** - 항상 커밋된 스냅샷만 읽는다

------------------------------------------------------------------------

### 2) Non-Repeatable Read

같은 행을 두 번 읽었는데 값이 바뀜

    A: SELECT salary = 1000
    B: UPDATE salary = 2000; COMMIT
    A: SELECT salary = 2000

오라클 READ COMMITTED: **발생함**\
이유: SELECT 시점 기준 스냅샷

------------------------------------------------------------------------

### 3) Phantom Read

조회 결과 행 개수가 바뀜

    A: SELECT COUNT(*) = 10
    B: INSERT; COMMIT
    A: SELECT COUNT(*) = 11

오라클 READ COMMITTED: **발생함**

------------------------------------------------------------------------

### 4) Lost Update (갱신 손실) ★실무 핵심

    재고 = 10

    A: SELECT 10
    B: SELECT 10
    A: UPDATE 9
    B: UPDATE 9
    → 결과 9 (정상은 8)

READ COMMITTED: **발생 가능**

해결:

``` sql
SELECT * FROM stock WHERE id=1 FOR UPDATE;
```

------------------------------------------------------------------------

### 5) Write Skew

서로 다른 행을 수정했지만 규칙 깨짐

예: 최소 1명 당직 규칙 - 의사1 off - 의사2 off → 당직 0명

READ COMMITTED: **발생 가능** → SERIALIZABLE 필요

------------------------------------------------------------------------

### 6) Dirty Write

커밋되지 않은 데이터를 덮어씀

오라클: **절대 발생 안함** - UPDATE 시 행 배타락이 자동 생성됨

------------------------------------------------------------------------

## 4. READ COMMITTED (기본 격리수준)

특징 - 쿼리 실행 시점 기준 스냅샷 - 성능 좋음 - 대부분 시스템 사용

  이상현상        발생
  --------------- ------
  Dirty Read      X
  NonRepeatable   O
  Phantom         O
  Lost Update     O
  Write Skew      O
  Dirty Write     X

------------------------------------------------------------------------

## 5. SERIALIZABLE

오라클은 락으로 막지 않고 **충돌 감지 방식** 사용

충돌 시:

    ORA-08177: can't serialize access for this transaction

→ 트랜잭션 실패 후 재시도 필요

  이상현상        발생
  --------------- ------
  Dirty Read      X
  NonRepeatable   X
  Phantom         X
  Lost Update     X
  Write Skew      X
  Dirty Write     X

------------------------------------------------------------------------

## 6. 실무 사용 가이드

### 일반 조회

READ COMMITTED 그대로 사용

### 수정 화면 (데이터 편집)

``` sql
SELECT ... FOR UPDATE
```

→ 다른 사용자 수정 방지 (비관락)

### 예약 / 좌석 / 재고 / 결제

낙관락 (version 컬럼)

``` sql
UPDATE product
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = :oldVersion;
```

### 회계 마감 / 금융

SERIALIZABLE

------------------------------------------------------------------------

## 7. 핵심 결론

-   오라클 SELECT는 락이 없다 (MVCC)
-   격리수준은 읽기 문제 해결용
-   Lost Update는 격리수준으로 해결 불가
-   반드시 `FOR UPDATE` 또는 `version` 필요

**정리** - Isolation Level → 무엇을 볼 수 있나(읽기) - Lock / Version →
무엇을 쓸 수 있나(쓰기)
