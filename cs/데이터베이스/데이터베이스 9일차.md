# 트랜잭션 격리수준 & 동시성 이상현상 가이드

## 1. 왜 격리수준이 존재하는가

여러 사용자가 동시에 데이터베이스를 조회하고 수정하면\
예상하지 못한 데이터 불일치가 발생할 수 있다.\
데이터베이스는 각 트랜잭션이 다른 트랜잭션의 변경을 **어느 정도까지 볼
수 있는지**를 제어하기 위해\
**격리수준(Isolation Level)** 을 사용한다.

격리수준은 트레이드오프 관계이다:

  높은 격리수준   낮은 격리수준
  --------------- ----------------
  높은 일관성     높은 성능
  더 많은 락      더 많은 동시성

------------------------------------------------------------------------

## 2. 6가지 주요 동시성 이상현상

### 1) Dirty Read (더티 리드)

다른 트랜잭션이 **커밋하지 않은 데이터**를 읽는 현상.

예시\
A가 잔액을 수정 → 아직 커밋 안함\
B가 그 잔액을 읽음\
A가 롤백

B는 실제로 존재하지 않았던 값을 읽은 것이다.

------------------------------------------------------------------------

### 2) Non-Repeatable Read (반복 불가능 읽기)

하나의 트랜잭션 안에서 같은 행을 두 번 읽었는데 **값이 달라지는 현상**.

A가 가격 100 조회\
B가 가격을 80으로 수정 후 커밋\
A가 다시 조회 → 80

------------------------------------------------------------------------

### 3) Phantom Read (팬텀 리드)

같은 쿼리를 다시 실행했는데 **행의 개수가 달라지는 현상**.

A: 오늘 예약 조회 → 10행\
B: 예약 추가 후 커밋\
A: 다시 조회 → 11행

------------------------------------------------------------------------

### 4) Lost Update (갱신 손실)

두 트랜잭션이 같은 값을 읽고 서로 덮어써서 한쪽 수정이 사라지는 현상.

재고 = 10

A가 10 조회\
B가 10 조회\
A가 9로 수정\
B가 9로 수정

정상 결과는 8이어야 한다.

------------------------------------------------------------------------

### 5) Write Skew (쓰기 왜곡)

서로 다른 행을 수정했지만 **비즈니스 규칙이 깨지는 현상**.

규칙: 항상 최소 1명의 의사는 당직이어야 한다.

의사 A 당직 해제\
의사 B 당직 해제\
두 트랜잭션 모두 성공 → 당직 의사 0명

------------------------------------------------------------------------

### 6) Dirty Write (가장 위험)

다른 트랜잭션이 **아직 커밋하지 않은 데이터 위에 덮어써버리는 현상**.

A가 잔액 수정 → 커밋 안함\
B가 같은 잔액 수정 후 커밋\
A가 롤백

데이터베이스는 원래 상태로 복구할 수 없게 된다.

**모든 DBMS는 격리수준과 상관없이 Dirty Write는 항상 방지한다.**

------------------------------------------------------------------------

## 3. 격리수준

### READ UNCOMMITTED

-   읽기 락 없음
-   최고 성능
-   거의 일관성 보장 없음

### READ COMMITTED (오라클 기본)

-   커밋된 데이터만 읽을 수 있음
-   MVCC 스냅샷 읽기 사용

### REPEATABLE READ (MySQL 기본)

-   한번 읽은 행의 값이 트랜잭션 종료까지 유지됨
-   행 값 변경 방지

### SERIALIZABLE

-   트랜잭션이 순차적으로 실행된 것처럼 동작
-   가장 강한 일관성
-   가장 낮은 성능

------------------------------------------------------------------------

## 4. 격리수준별 방지 가능한 이상현상

  ----------------------------------------------------------------------------
  이상현상         Read           Read Committed Repeatable     Serializable
                   Uncommitted                   Read           
  ---------------- -------------- -------------- -------------- --------------
  Dirty Read       방지 못함      방지           방지           방지

  Non-Repeatable   발생           발생           방지           방지
  Read                                                          

  Phantom Read     발생           발생           발생           방지

  Lost Update      발생           발생           부분 방지      방지

  Write Skew       발생           발생           발생           방지

  Dirty Write      항상 방지      항상 방지      항상 방지      항상 방지
  ----------------------------------------------------------------------------

------------------------------------------------------------------------

## 5. 실무 사용 예시

  시나리오         권장 방법
  ---------------- -----------------------------------
  일반 조회        Read Committed
  수정 화면        SELECT FOR UPDATE
  예약/재고/결제   낙관적 락(version) 또는 비관적 락
  금융 핵심 처리   Serializable

------------------------------------------------------------------------

## 6. 핵심 이해

격리수준은 주로 **읽기 일관성(read consistency)** 을 제어한다.

Lost Update 같은 쓰기 충돌 문제는 완전히 해결하지 못한다.\
쓰기 충돌 해결 방법:

-   비관적 락 (행 락)
-   낙관적 락 (버전 컬럼)

------------------------------------------------------------------------

## 최종 개념

**격리수준 = 무엇을 볼 수 있는가(읽기 제어)**\
**락/버전 = 무엇을 쓸 수 있는가(쓰기 제어)**
