# Oracle Undo & Redo 상세 정리

> 핵심 한 줄\
> **Undo = 과거로 되돌리기(논리 복구 + 읽기 스냅샷)**\
> **Redo = 미래로 다시 실행하기(장애 복구 + 커밋 보장)**

------------------------------------------------------------------------

## 1. UPDATE 실행 시 내부 동작 순서

예시:

``` sql
UPDATE account SET balance = 0 WHERE id = 1;
```

오라클은 절대 바로 디스크에 덮어쓰지 않는다.

### 1) Undo 기록

현재 값:

    balance = 10000

Undo 세그먼트에 저장:

    (id=1, 이전값=10000)

목적: - ROLLBACK 가능 - 읽기 일관성 제공(MVCC)

------------------------------------------------------------------------

### 2) Buffer Cache에서 데이터 수정

디스크가 아니라 메모리(SGA)의 Buffer Cache에서 수정된다.

    balance = 0

※ 아직 확정 데이터 아님 (uncommitted data)

------------------------------------------------------------------------

### 3) Redo 로그 기록

Redo Log Buffer에 변경 기록 생성:

    10000 → 0 변경 기록

목적: - 서버 장애 시 복구 가능하게

------------------------------------------------------------------------

### 4) Row Lock 생성

행에 TX Lock 생성:

    이 행은 현재 트랜잭션이 사용 중

목적: - Dirty Write 방지

------------------------------------------------------------------------

## 2. COMMIT 시 실제 동작

``` sql
COMMIT;
```

많이 착각하는 것: \> COMMIT = 디스크에 데이터 쓰기 ❌

실제 동작: **Redo 로그만 디스크에 기록(LGWR flush)**

데이터 블록은 나중(DBWR)이 써도 된다.

이게 가능한 이유: \> Redo 로그만 있으면 언제든 복구 가능하기 때문

------------------------------------------------------------------------

## 3. ROLLBACK 시 동작

``` sql
ROLLBACK;
```

Undo 사용:

    이전값 10000 복구

  작업       사용하는 것
  ---------- -------------
  COMMIT     Redo
  ROLLBACK   Undo

------------------------------------------------------------------------

## 4. MVCC에서 Undo의 역할 (매우 중요)

다른 세션:

``` sql
SELECT balance FROM account WHERE id=1;
```

트랜잭션이 아직 커밋 안됐어도 SELECT는 대기하지 않는다.

이유: 현재 블록을 읽는 것이 아니라 Undo의 과거 버전을 읽는다.

즉 Undo는 단순 롤백용이 아니라: **읽기 일관성(Read Consistency) 제공
장치**

그래서 오라클 특징: - SELECT는 안 막힘 - UPDATE는 막힘

------------------------------------------------------------------------

## 5. 장애 발생 시 (Crash Recovery)

상황: - UPDATE 후 COMMIT - 데이터파일 기록 전 서버 다운

데이터 파일:

    balance = 10000

Redo 로그:

    10000 → 0 기록 존재

재기동 시: **SMON이 Redo 적용(roll forward)**

→ 데이터 복구 완료

------------------------------------------------------------------------

## 6. 구성요소 역할

  구성요소       역할
  -------------- ----------------
  Undo           과거 상태 저장
  Redo           변경 재실행
  Buffer Cache   작업 메모리
  Row Lock       동시 수정 방지
  SCN            시간 기준점

------------------------------------------------------------------------

## 7. Undo vs Redo 비교

  구분        Undo                      Redo
  ----------- ------------------------- -------------------
  목적        되돌리기                  재실행
  사용 시점   ROLLBACK / SELECT(MVCC)   COMMIT / 장애복구
  저장 내용   이전 데이터               변경 기록
  없으면      롤백 불가 / 조회 깨짐     DB 복구 불가

------------------------------------------------------------------------

## 최종 이해

-   Undo → 이전 상태로 돌아가기 위한 타임머신
-   Redo → 장애 후에도 다시 살아나게 하는 블랙박스

오라클은 이를 통해: **동시성(MVCC) + 안정성(Recovery)을 동시에
보장한다**
