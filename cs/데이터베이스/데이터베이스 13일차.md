# 동시성 · 트랜잭션 · SERIALIZABLE 총정리 (Spring + MyBatis 기준)

> 핵심 한 줄\
> **DB는 SQL 한 문장은 안전하게 실행하지만, 여러 문장으로 이루어진
> 비즈니스 로직의 정합성은 자동으로 보장하지 않는다.**\
> 그래서 격리수준, 락, AOP, 재시도 전략이 필요하다.

------------------------------------------------------------------------

## 1. 트랜잭션 기본 개념

트랜잭션(Transaction)이란: - 여러 SQL을 하나의 작업 단위로 묶어 - 모두
성공하면 커밋(commit) - 중간에 문제 발생하면 롤백(rollback)

### ACID

  항목          의미
  ------------- ----------------------------------
  Atomicity     전부 성공 or 전부 실패
  Consistency   규칙을 깨지 않음
  Isolation     동시에 실행돼도 서로 간섭 최소화
  Durability    커밋 후 영구 저장

------------------------------------------------------------------------

## 2. Spring에서 트랜잭션 (@Transactional) 동작

``` java
@Service
public class OrderService {

    @Transactional
    public void placeOrder(){
        orderMapper.insert(...);
        stockMapper.decrease(...);
    }
}
```

실제 실행 흐름:

    Controller
      ↓
    Spring AOP Proxy
      ↓
    TransactionInterceptor
      ↓
    DB Connection 획득
      ↓
    SQL 실행 (MyBatis)
      ↓
    메서드 종료
      ↓
    commit / rollback

### 핵심

-   트랜잭션은 메서드 **안에 있는 것이 아니라 메서드를 감싸는 프록시
    밖에서 시작/종료**
-   커밋은 **메서드 종료 시점**에 발생

------------------------------------------------------------------------

## 3. AOP란?

AOP(Aspect Oriented Programming):

> 메서드 실행을 가로채서 앞뒤로 코드를 자동 실행하는 기술

예: - 트랜잭션 - 로깅 - 권한 검사 - 캐싱

``` text
Proxy
  BEGIN TRANSACTION
  ↓
  실제 메서드 실행
  ↓
  COMMIT
```

## 4. MVCC (다중 버전 동시성 제어)

DB는 데이터를 바로 덮어쓰지 않는다.

UPDATE 시 실제 동작:

1)  Undo 저장 (과거 값 보관)
2)  메모리에서 값 변경
3)  Redo 로그 기록

그래서 SELECT는 락 없이 과거 스냅샷을 읽는다.

------------------------------------------------------------------------

## 5. 격리수준 (Isolation Level)

### READ COMMITTED (기본)

-   SELECT: 락 없음
-   UPDATE: 행락(Row Lock)

``` sql
SELECT * FROM account WHERE id=1;
```

다른 트랜잭션이 수정 중이어도 읽을 수 있음.

### REPEATABLE READ

-   트랜잭션 시작 시점의 스냅샷 유지
-   읽은 행 값은 변하지 않게 보임

### SERIALIZABLE

> 결과가 반드시 순차 실행과 동일하도록 강제

동작: - 읽은 조건 범위 감시 - 커밋 시 논리 검사 - 모순이면 롤백

------------------------------------------------------------------------

## 6. Write Skew (가장 위험한 동시성 버그)

초기 상태:

    의사 A: 근무(Y)
    의사 B: 근무(Y)
    (최소 1명 필요)

서비스 코드:

``` java
@Transactional
public void leaveWork(long id){
    int count = mapper.countOnCall();
    if(count <= 1) throw new RuntimeException();
    mapper.offDuty(id);
}
```

동시 실행:

T1:

``` sql
SELECT COUNT(*) = 2
UPDATE A → N
```

T2:

``` sql
SELECT COUNT(*) = 2
UPDATE B → N
```

결과:

    Y = 0명

DB 에러 없음 → 논리 버그

------------------------------------------------------------------------

## 7. SERIALIZABLE 내부 원리 (SSI)

DB가 기록하는 것:

-   read-set (읽은 조건)
-   write-set (수정한 데이터)

예:

``` sql
SELECT COUNT(*) FROM doctor WHERE on_call='Y';
```

의미:

    "on_call='Y' 상태를 근거로 판단했다"

다른 트랜잭션이 이 조건을 바꾸면:

    T1 → T2 (시간 순서 제약 생성)

만약:

    T1 → T2
    T2 → T1

사이클 발생 → 직렬 실행 불가능 → 롤백

------------------------------------------------------------------------

## 8. Row Lock과 SELECT FOR UPDATE

행락이 생기는 경우:

-   UPDATE
-   DELETE
-   SELECT FOR UPDATE

``` sql
SELECT * FROM doctor WHERE id=1 FOR UPDATE;
```

효과: - 해당 행 트랜잭션 종료까지 잠금 - 다른 UPDATE 대기

### 실무 패턴

    READ COMMITTED + SELECT FOR UPDATE

→ SERIALIZABLE 대신 많이 사용

### 9.1 CREATE --- 중복 INSERT (Race Condition)

#### (1) 취약한 구현: `SELECT 존재확인 → INSERT`

**Mapper(XML)**

``` xml
<select id="existsUser" resultType="int">
  SELECT COUNT(*)
  FROM users
  WHERE id = #{id}
</select>

<insert id="insertUser">
  INSERT INTO users(id, name)
  VALUES(#{id}, #{name})
</insert>
```

**Service**

``` java
@Transactional
public void createUser(long id, String name){
    int exists = userMapper.existsUser(id);
    if(exists > 0) throw new IllegalStateException("이미 존재");

    // ⚠️ 여기 사이에 다른 트랜잭션이 끼어들면 중복 INSERT 발생
    userMapper.insertUser(id, name);
}
```

**동시 실행 타임라인** - T1: exists=0 확인 - T2: exists=0 확인 - T1:
INSERT 성공 - T2: INSERT 시도 → **PK UNIQUE 위반**

**핵심** - `SELECT`는 락이 없고(대부분 MVCC 스냅샷 읽기) - `SELECT`와
`INSERT` 사이에 **Gap** 이 생기기 때문

#### (2) 실무 해결 1: "그냥 INSERT하고 예외 처리"

-   DB 유일성은 PK/UNIQUE가 보장
-   선조회(`exists`)로 막지 말고 INSERT 먼저 시도
-   중복키는 **409 Conflict** 등으로 변환

> Spring-MyBatis 조합에서 중복키는 보통 `DuplicateKeyException` 계열로
> 번역됨

#### (3) 실무 해결 2: UPSERT (DB별 문법)

-   Oracle: `MERGE`
-   PostgreSQL: `INSERT ... ON CONFLICT`
-   MySQL: `INSERT ... ON DUPLICATE KEY UPDATE`

### 9.2 READ --- Non-repeatable / Phantom (조회 결과 변동)

#### (1) Non-repeatable read (READ COMMITTED 특징)

**Mapper**

``` xml
<select id="selectBalance" resultType="int">
  SELECT balance
  FROM account
  WHERE id = #{id}
</select>
```

**Service**

``` java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void demoNonRepeatable(long id){
    int v1 = accountMapper.selectBalance(id); // 예: 10000

    // (여기서 다른 트랜잭션이 UPDATE+COMMIT 하면)

    int v2 = accountMapper.selectBalance(id); // 예: 0 (같은 TX인데 바뀜)
}
```

#### (2) Phantom read (조건 결과집합 변화)

예약 건수 예시.

**Mapper**

``` xml
<select id="countReservationByDate" resultType="int">
  SELECT COUNT(*)
  FROM reservation
  WHERE resv_date = #{date}
</select>
```

**Service**

``` java
@Transactional(isolation = Isolation.READ_COMMITTED)
public int countTwice(String date){
    int c1 = reservationMapper.countReservationByDate(date); // 9

    // (다른 트랜잭션이 같은 date로 INSERT+COMMIT 하면)

    int c2 = reservationMapper.countReservationByDate(date); // 10
    return c2;
}
```

**정리** - "조회 결과"를 근거로 비즈니스 판단하면 흔들릴 수 있음 - 특히
**COUNT / 존재확인 / 조건 검색** 기반 로직에서 자주 터짐

### 9.3 UPDATE --- Lost Update (덮어쓰기)

#### (1) 취약한 구현: `SELECT로 읽고 → 계산해서 → UPDATE`

**Mapper**

``` xml
<select id="selectBalance" resultType="int">
  SELECT balance
  FROM account
  WHERE id = #{id}
</select>

<update id="updateBalance">
  UPDATE account
  SET balance = #{balance}
  WHERE id = #{id}
</update>
```

**Service**

``` java
@Transactional
public void withdrawBad(long id, int amount){
    int bal = accountMapper.selectBalance(id); // T1,T2가 둘 다 10000 읽을 수 있음
    int newBal = bal - amount;
    accountMapper.updateBalance(id, newBal);   // 마지막 커밋이 이김(덮어쓰기)
}
```

#### (2) 해결 1: 비관적 락 (SELECT FOR UPDATE)

**Mapper**

``` xml
<select id="selectAccountForUpdate" resultType="Account">
  SELECT id, balance
  FROM account
  WHERE id = #{id}
  FOR UPDATE
</select>

<update id="decreaseBalance">
  UPDATE account
  SET balance = balance - #{amount}
  WHERE id = #{id}
</update>
```

**Service**

``` java
@Transactional
public void withdrawPessimistic(long id, int amount){
    accountMapper.selectAccountForUpdate(id); // row lock 획득 (TX 끝날 때까지 유지)

    int rows = accountMapper.decreaseBalance(id, amount);
    if(rows != 1) throw new IllegalStateException("갱신 실패");
}
```

#### (3) 해결 2: 낙관적 락 (version)

> MyBatis는 JPA처럼 자동 낙관락이 없어서, **직접 version 컬럼을 WHERE에
> 넣어야 함**

**Mapper**

``` xml
<select id="selectAccountWithVersion" resultType="Account">
  SELECT id, balance, version
  FROM account
  WHERE id = #{id}
</select>

<update id="updateBalanceOptimistic">
  UPDATE account
  SET balance = #{balance},
      version = version + 1
  WHERE id = #{id}
    AND version = #{version}
</update>
```

**Service**

``` java
@Transactional
public void withdrawOptimistic(long id, int amount){
    Account a = accountMapper.selectAccountWithVersion(id);

    int newBal = a.getBalance() - amount;
    int rows = accountMapper.updateBalanceOptimistic(id, newBal, a.getVersion());

    if(rows == 0){
        // 누군가 먼저 갱신해서 version이 변함
        throw new OptimisticLockingFailureException("동시 수정 감지");
    }
}
```

### 9.4 UPDATE --- Write Skew (조건 기반 로직 붕괴)

#### (1) 취약한 구현: "조건으로 판단 → 다른 행 업데이트"

당직 최소 1명 규칙 예시.

**Mapper**

``` xml
<select id="countOnCall" resultType="int">
  SELECT COUNT(*)
  FROM doctor
  WHERE on_call = 'Y'
</select>

<update id="offDuty">
  UPDATE doctor
  SET on_call = 'N'
  WHERE id = #{id}
</update>
```

**Service**

``` java
@Transactional
public void leaveWorkBad(long doctorId){
    int count = doctorMapper.countOnCall(); // T1,T2가 동시에 2를 읽을 수 있음
    if(count <= 1) throw new IllegalStateException("마지막 당직자 퇴근 불가");

    doctorMapper.offDuty(doctorId);         // 서로 다른 행을 꺼버려서 0명 가능
}
```

**왜 DB가 못 막나?** - T1은 id=1만 수정, T2는 id=2만 수정 → **행 충돌
없음** - 하지만 "on_call='Y' 최소 1명"이라는 **조건 무결성**이 깨짐

#### (2) 실무 해결 1(권장): '규칙 행' 잠금 (SELECT FOR UPDATE)

조건 집합을 직접 잠그기 어렵기 때문에, **규칙을 대표하는 단일 행을
잠가서 직렬화**한다.

**테이블 예시**

``` sql
CREATE TABLE rule_lock (
  rule_key VARCHAR2(30) PRIMARY KEY
);

INSERT INTO rule_lock(rule_key) VALUES ('ONCALL_MIN1');
```

**Mapper**

``` xml
<select id="lockRule" resultType="int">
  SELECT 1
  FROM rule_lock
  WHERE rule_key = 'ONCALL_MIN1'
  FOR UPDATE
</select>
```

**Service**

``` java
@Transactional
public void leaveWorkWithRuleLock(long doctorId){
    doctorMapper.lockRule(); // 여기서 직렬화 (다른 트랜잭션은 대기)

    int count = doctorMapper.countOnCall();
    if(count <= 1) throw new IllegalStateException("마지막 당직자 퇴근 불가");

    doctorMapper.offDuty(doctorId);
}
```

#### (3) 실무 해결 2: SERIALIZABLE + 재시도

``` java
@Transactional(isolation = Isolation.SERIALIZABLE)
public void leaveWorkSerializable(long doctorId){
    int count = doctorMapper.countOnCall();
    if(count <= 1) throw new IllegalStateException("마지막 당직자 퇴근 불가");
    doctorMapper.offDuty(doctorId);
}
```

-   장점: 논리적으로 가장 강함
-   단점: 커밋 시점에 직렬화 실패(예: ORA-08177) 가능 → 짧은 재시도 필요

### 9.5 DELETE --- 존재 확인 후 처리 버그 (0 rows 업데이트)

#### 취약한 구현

**Mapper**

``` xml
<select id="selectUser" resultType="User">
  SELECT id, name
  FROM users
  WHERE id = #{id}
</select>

<update id="updateUserName">
  UPDATE users
  SET name = #{name}
  WHERE id = #{id}
</update>

<delete id="deleteUser">
  DELETE FROM users
  WHERE id = #{id}
</delete>
```

**Service**

``` java
@Transactional
public void renameUserBad(long id, String newName){
    User u = userMapper.selectUser(id); // 존재 확인
    if(u == null) throw new IllegalStateException("없음");

    // (이 사이에 다른 트랜잭션이 deleteUser(id) + commit)

    int rows = userMapper.updateUserName(id, newName);
    // ⚠️ rows == 0 인데도 성공 처리하면 논리 버그
}
```

#### 해결 1: row count 체크

``` java
@Transactional
public void renameUserSafe(long id, String newName){
    int rows = userMapper.updateUserName(id, newName);
    if(rows == 0){
        throw new IllegalStateException("대상 없음(동시 삭제 가능)");
    }
}
```

#### 해결 2: 강한 보장이 필요하면 FOR UPDATE

**Mapper**

``` xml
<select id="selectUserForUpdate" resultType="User">
  SELECT id, name
  FROM users
  WHERE id = #{id}
  FOR UPDATE
</select>
```

**Service**

``` java
@Transactional
public void renameUserWithLock(long id, String newName){
    User u = userMapper.selectUserForUpdate(id); // 행락 획득
    if(u == null) throw new IllegalStateException("없음");

    userMapper.updateUserName(id, newName);
}
```

                                                                            version

                                                                            Serializable

                                                                            FOR UPDATE

## 10. 격리수준별 특징 비교

  항목         READ COMMITTED   REPEATABLE READ   SERIALIZABLE
  ------------ ---------------- ----------------- --------------
  읽기 락      없음             없음              조건 감시
  행락         쓰기만           쓰기만            쓰기만
  Write Skew   발생             발생              방지
  충돌 처리    대기             대기              커밋 실패

## 11. 실무 설계 방법

### 방법 1: SELECT FOR UPDATE

``` sql
SELECT * FROM rule WHERE rule_key='ONCALL' FOR UPDATE;
```

→ 규칙을 대표하는 행을 잠금

### 방법 2: 낙관적 락

``` sql
UPDATE product
SET qty = qty - 1,
    version = version + 1
WHERE id=1 AND version=3;
```

0 rows → 재시도

### 방법 3: SERIALIZABLE + Retry

``` java
@Retryable
@Transactional(isolation = Isolation.SERIALIZABLE)
public void reserve(){ ... }
```

## 12. 최종 핵심 정리

-   트랜잭션 문제는 DB 문제가 아니라 **로직 설계 문제**
-   SELECT → 판단 → UPDATE 구조가 충돌의 원인
-   SERIALIZABLE은 락이 아니라 **논리 검증**
-   SELECT FOR UPDATE는 **물리적 차단**
-   대부분 실무는:

### 기억해야 할 한 문장

> DB는 데이터를 보호하지만, 비즈니스 규칙은 보호하지 않는다.
