# MPA / SPA / CSR / SSR 개념 정리

웹 구조 및 렌더링 방식을 **두 관점**으로 정리

---

## 1️⃣ 페이지 구조 기준

| 개념 | 의미 | 동작 방식 | 예시 |
|------|------|----------|------|
| MPA (Multi Page Application) | 여러 개의 HTML 페이지 | 페이지 이동 시 서버에서 **새 HTML** 로드 | 네이버, 관공서 사이트 |
| SPA (Single Page Application) | 단 하나의 HTML 페이지 | 새로고침 없이 JS로 **컴포넌트 전환** | React/Vue 기본 앱, Gmail |

### 비유
- **MPA**: 메뉴 바뀌면 **새 접시**
- **SPA**: **같은 접시**에 음식만 교체

---

## 2️⃣ 렌더링 위치 기준

| 개념 | 의미 | 동작 방식 | 예시 |
|------|------|----------|------|
| CSR (Client Side Rendering) | 브라우저에서 렌더링 | 빈 HTML + JS → 브라우저가 화면 생성 | React, Vue 기본 방식 |
| SSR (Server Side Rendering) | 서버에서 렌더링 | 서버에서 완성 HTML 생성 → 전송 | JSP/JSTL, PHP, Next.js SSR |

### 비유
- **CSR**: 셀프바 (브라우저가 요리)
- **SSR**: 요리사가 만들어 바로 제공 (서버가 렌더링)

---

## 3️⃣ 조합 가능성

| 조합 | 가능 여부 | 대표 예시 |
|------|---------|-----------|
| MPA + SSR | 가능 | 기존 웹 서비스(JSP, Spring MVC + Thymeleaf) |
| SPA + CSR | 가능(가장 흔함) | CRA/Vite 기반 React 앱 |
| SPA + SSR | 가능 | Next.js (초기 SSR → 이후 SPA처럼 동작) |
| MPA + CSR | 가능 | 각 페이지에 React 일부 적용한 레거시 |

> SPA ⊆ CSR (대부분의 SPA는 CSR 기반)  
> 하지만 **SPA = CSR**은 아님 (별개 개념)

---

## 4️⃣ 언제 어떤 방식이 좋은가?

| 상황 | 추천 방식 |
|------|---------|
| 로그인 기반 대시보드, 내부 서비스 | SPA + CSR |
| SEO 중요(블로그, 쇼핑몰) | MPA + SSR 또는 SPA + SSR |
| 초기 속도 + SEO + 최신 기술 모두 필요 | Next.js (SSR/SSG/CSR 혼합) |
