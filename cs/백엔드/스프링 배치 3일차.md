# Spring Batch 03. 실무 운영, 재시작, 트랜잭션, 스케줄링

## 1. 실무 배치에서 중요한 것

Spring Batch를 코드로 실행하는 것보다 더 중요한 것은 운영 안정성이다.

실무에서 배치는 다음 문제를 반드시 고려해야 한다.

```text
중복 실행 방지
실패 시 재시작
대량 데이터 성능
트랜잭션 범위
락 경합
외부 시스템 연계 실패
로그와 모니터링
스케줄링
운영자 수동 재실행
```

## 2. 배치 실행 방식

```text
애플리케이션 시작 시 자동 실행
REST API로 실행
스케줄러로 실행
CommandLineRunner로 실행
Jenkins에서 실행
Kubernetes CronJob으로 실행
사내 배치 관리 시스템에서 실행
```

운영에서는 보통 자동 실행을 꺼둔다.

```yaml
spring:
  batch:
    job:
      enabled: false
```

그리고 원하는 시점에 JobLauncher로 실행한다.

## 3. 스케줄러로 실행하기

```java
@Configuration
@EnableScheduling
@RequiredArgsConstructor
public class BatchScheduler {

    private final JobLauncher jobLauncher;
    private final Job dormantMemberJob;

    @Scheduled(cron = "0 0 2 * * *")
    public void runDormantMemberJob() throws Exception {
        JobParameters jobParameters = new JobParametersBuilder()
                .addString("baseDate", LocalDate.now().toString())
                .toJobParameters();

        jobLauncher.run(dormantMemberJob, jobParameters);
    }
}
```

### 코드 설명

```java
@EnableScheduling
```

Spring 스케줄링 기능을 켠다.

```java
@Scheduled(cron = "0 0 2 * * *")
```

매일 새벽 2시에 실행한다.

```java
.addString("baseDate", LocalDate.now().toString())
```

오늘 날짜를 JobParameter로 넣는다.
같은 날짜로 이미 성공한 Job은 다시 실행되지 않을 수 있다.

## 4. 수동 실행 API

```java
@RestController
@RequiredArgsConstructor
@RequestMapping("/admin/batch")
public class BatchAdminController {

    private final JobLauncher jobLauncher;
    private final Job dormantMemberJob;

    @PostMapping("/dormant-members")
    public String run(@RequestParam String baseDate) throws Exception {
        JobParameters jobParameters = new JobParametersBuilder()
                .addString("baseDate", baseDate)
                .toJobParameters();

        JobExecution execution = jobLauncher.run(dormantMemberJob, jobParameters);

        return "jobExecutionId=" + execution.getId()
                + ", status=" + execution.getStatus();
    }
}
```

운영 API는 반드시 권한 검사가 필요하다.

```text
관리자 인증
IP 제한
중복 실행 방지
실행 이력 조회
실패 시 알림
```

## 5. Job 중복 실행 방지

같은 JobParameters로 이미 완료된 Job을 다시 실행하면 Spring Batch는 재실행을 막을 수 있다.
하지만 동시에 두 요청이 들어오는 상황은 별도로 고려해야 한다.

대표적인 방법:

```text
JobParameters를 명확히 설계
JobExplorer로 실행 중 Job 확인
DB Lock 사용
분산락 사용
스케줄러 서버를 1대로 제한
Kubernetes CronJob concurrencyPolicy 사용
```

## 6. JobExplorer로 실행 중 Job 확인

```java
@Service
@RequiredArgsConstructor
public class BatchRunService {

    private final JobExplorer jobExplorer;
    private final JobLauncher jobLauncher;
    private final Job dormantMemberJob;

    public void runDormantMemberJob(String baseDate) throws Exception {
        Set<JobExecution> runningExecutions =
                jobExplorer.findRunningJobExecutions("dormantMemberJob");

        if (!runningExecutions.isEmpty()) {
            throw new IllegalStateException("이미 실행 중인 Job이 있습니다.");
        }

        JobParameters jobParameters = new JobParametersBuilder()
                .addString("baseDate", baseDate)
                .toJobParameters();

        jobLauncher.run(dormantMemberJob, jobParameters);
    }
}
```

```java
jobExplorer.findRunningJobExecutions("dormantMemberJob")
```

현재 실행 중인 JobExecution을 조회한다.

단, 여러 서버에서 동시에 호출하면 이 체크만으로 완벽한 락은 아니다.
그럴 때는 DB Lock 또는 분산락을 고려한다.

## 7. 재시작 개념

Spring Batch의 강점 중 하나는 실패한 배치를 재시작할 수 있다는 것이다.

예를 들어 1,000건 처리 중 600건까지 처리하고 실패했다고 하자.

```text
1~100 commit
101~200 commit
201~300 commit
301~400 commit
401~500 commit
501~600 commit
601~700 처리 중 실패
```

재시작하면 이미 commit된 chunk 이후부터 다시 처리할 수 있다.
이 기능이 동작하려면 Reader와 ExecutionContext 설정이 중요하다.

## 8. allowStartIfComplete

기본적으로 완료된 Step은 다시 실행되지 않는다.
완료된 Step도 매번 실행하고 싶다면 다음 옵션을 사용할 수 있다.

```java
@Bean
public Step alwaysRunStep(JobRepository jobRepository,
                          PlatformTransactionManager transactionManager) {
    return new StepBuilder("alwaysRunStep", jobRepository)
            .tasklet((contribution, chunkContext) -> {
                System.out.println("항상 실행되는 Step");
                return RepeatStatus.FINISHED;
            }, transactionManager)
            .allowStartIfComplete(true)
            .build();
}
```

사용 예:

```text
매번 임시 파일 정리
매번 로그 초기화
매번 사전 검증
```

완료된 Step도 다시 실행되므로 멱등성이 필요하다.

## 9. startLimit

특정 Step의 실행 횟수를 제한할 수 있다.

```java
@Bean
public Step limitedStep(JobRepository jobRepository,
                        PlatformTransactionManager transactionManager) {
    return new StepBuilder("limitedStep", jobRepository)
            .tasklet((contribution, chunkContext) -> {
                System.out.println("최대 3번만 실행 가능");
                return RepeatStatus.FINISHED;
            }, transactionManager)
            .startLimit(3)
            .build();
}
```

3번 넘게 실행하면 예외가 발생한다.

## 10. 트랜잭션과 chunk size

```java
.<Member, Member>chunk(100, transactionManager)
```

이 코드는 100건마다 commit한다.

chunk size가 너무 작으면:

```text
commit 횟수 증가
DB round trip 증가
성능 저하
```

chunk size가 너무 크면:

```text
트랜잭션 시간이 길어짐
undo/redo 증가
락 유지 시간 증가
실패 시 롤백 부담 증가
메모리 사용량 증가
```

실무에서는 데이터 양, DB 성능, writer 방식에 따라 100, 500, 1000 정도부터 테스트한다.

## 11. 외부 API 호출 배치 주의점

```java
@Component
@RequiredArgsConstructor
public class ExternalApiWriter implements ItemWriter<Member> {

    private final ExternalMemberClient externalMemberClient;

    @Override
    public void write(Chunk<? extends Member> chunk) {
        for (Member member : chunk) {
            externalMemberClient.send(member);
        }
    }
}
```

DB 트랜잭션과 외부 API는 함께 롤백되지 않는다.

```text
1번 회원 API 전송 성공
2번 회원 API 전송 성공
3번 회원 API 전송 성공
4번 회원 API 전송 실패
chunk 실패
DB rollback
```

DB는 rollback되지만 이미 성공한 API 호출은 취소되지 않는다.

그래서 외부 연계 배치는 다음 설계가 중요하다.

```text
인터페이스 테이블 사용
상태값 관리
멱등키 사용
재전송 가능 구조
요청/응답 로그 저장
```

## 12. 인터페이스 테이블 패턴

PLM, ERP, MES 같은 시스템 연계에서는 인터페이스 테이블을 많이 둔다.

예시 테이블:

```text
IF_PRODUCT_SEND
```

| 컬럼          | 의미                 |
| ------------- | -------------------- |
| IF_ID         | 인터페이스 ID        |
| PRODUCT_ID    | 제품 ID              |
| STATUS        | READY, SUCCESS, FAIL |
| REQUEST_BODY  | 요청 데이터          |
| RESPONSE_BODY | 응답 데이터          |
| ERROR_MESSAGE | 오류 메시지          |
| CREATED_AT    | 생성일               |
| SENT_AT       | 전송일               |

흐름:

```text
1. PLM 데이터 조회
2. IF_PRODUCT_SEND에 READY 상태로 저장
3. 별도 배치가 READY 데이터를 외부 시스템으로 전송
4. 성공하면 SUCCESS
5. 실패하면 FAIL
6. FAIL 데이터는 재처리 가능
```

이 구조가 실무에서 안정적이다.

## 13. 상태 기반 재처리 예시

Reader SQL:

```sql
SELECT IF_ID, PRODUCT_ID, REQUEST_BODY
FROM IF_PRODUCT_SEND
WHERE STATUS IN ('READY', 'FAIL')
ORDER BY IF_ID
```

Writer:

```java
@Component
@RequiredArgsConstructor
public class ProductInterfaceWriter implements ItemWriter<ProductInterfaceData> {

    private final ProductApiClient productApiClient;
    private final ProductInterfaceMapper mapper;

    @Override
    public void write(Chunk<? extends ProductInterfaceData> chunk) {
        for (ProductInterfaceData data : chunk) {
            try {
                ApiResponse response = productApiClient.send(data);

                mapper.updateSuccess(
                        data.getIfId(),
                        response.getBody(),
                        LocalDateTime.now()
                );
            } catch (Exception e) {
                mapper.updateFail(
                        data.getIfId(),
                        e.getMessage(),
                        LocalDateTime.now()
                );
            }
        }
    }
}
```

한 건 실패가 전체 Step 실패로 이어지지 않게 할 수 있다.
다만 모든 실패를 삼켜버리면 운영자가 장애를 모를 수 있으므로 실패 건수 집계와 알림이 필요하다.

## 14. Oracle 기준 DB 락 주의점

Oracle에서는 update 시 row lock이 걸린다.
배치가 같은 row를 오래 잡고 있으면 다른 트랜잭션이 대기할 수 있다.

주의할 점:

```text
chunk size 과도하게 크게 잡지 않기
where 조건에 인덱스 고려
order by 기준 명확히 하기
대량 update 시간대 분리
온라인 업무와 같은 테이블 처리 시 락 경합 확인
undo/redo 증가 고려
```

## 15. 대량 처리 성능 튜닝 체크리스트

```text
Reader SQL에 적절한 인덱스가 있는가?
불필요한 컬럼을 읽고 있지 않은가?
정렬 기준이 명확한가?
chunk size가 너무 작거나 크지 않은가?
Writer가 건별 update만 하고 있지 않은가?
외부 API 호출 timeout이 설정되어 있는가?
skip/retry 정책이 적절한가?
실패 데이터를 추적할 수 있는가?
배치 실행 시간이 온라인 업무 피크와 겹치지 않는가?
```

Listener 예시:

```java
@Component
public class StepSummaryListener implements StepExecutionListener {

    @Override
    public ExitStatus afterStep(StepExecution stepExecution) {
        System.out.println("Step name = " + stepExecution.getStepName());
        System.out.println("readCount = " + stepExecution.getReadCount());
        System.out.println("writeCount = " + stepExecution.getWriteCount());
        System.out.println("filterCount = " + stepExecution.getFilterCount());
        System.out.println("skipCount = " + stepExecution.getSkipCount());
        System.out.println("status = " + stepExecution.getStatus());

        return stepExecution.getExitStatus();
    }
}
```

Step에 연결:

```java
@Bean
public Step dormantMemberStep(JobRepository jobRepository,
                              PlatformTransactionManager transactionManager,
                              ItemReader<Member> reader,
                              ItemProcessor<Member, Member> processor,
                              ItemWriter<Member> writer,
                              StepSummaryListener listener) {
    return new StepBuilder("dormantMemberStep", jobRepository)
            .<Member, Member>chunk(100, transactionManager)
            .reader(reader)
            .processor(processor)
            .writer(writer)
            .listener(listener)
            .build();
}
```

## 17. Job 흐름 제어

Step 결과에 따라 다음 Step을 다르게 실행할 수 있다.

```java
@Bean
public Job flowJob(JobRepository jobRepository,
                   Step validateStep,
                   Step processStep,
                   Step failNotifyStep) {
    return new JobBuilder("flowJob", jobRepository)
            .start(validateStep)
                .on("FAILED").to(failNotifyStep)
            .from(validateStep)
                .on("*").to(processStep)
            .end()
            .build();
}
```

설명:

```java
.start(validateStep)
```

먼저 검증 Step을 실행한다.

```java
.on("FAILED").to(failNotifyStep)
```

검증 Step이 실패하면 실패 알림 Step으로 간다.

```java
.from(validateStep).on("*").to(processStep)
```

그 외 상태면 처리 Step으로 간다.

## 19. PLM to ERP 실무 예시

권장 구조:


```text
PRODUCT
```

```text
READY
SENDING
SUCCESS
FAIL
```

## 20. 운영에서 피해야 할 패턴

```text
JobParameters 없이 매번 실행
모든 예외를 catch하고 무시
외부 API 호출 결과를 저장하지 않음
대량 update를 한 트랜잭션으로 처리
Reader SQL에 order by 없음
실패한 데이터와 성공한 데이터 구분 없음
배치 로그가 System.out.println만 있음
chunk size를 근거 없이 크게 설정
운영 API에 인증 없음
```

## 21. 좋은 배치 설계 원칙

```text
재실행 가능해야 한다.
중복 실행되어도 데이터가 깨지지 않아야 한다.
실패 지점을 추적할 수 있어야 한다.
성공/실패 건수를 확인할 수 있어야 한다.
외부 시스템 연계는 상태 기반으로 설계해야 한다.
대량 DB 작업은 chunk와 인덱스를 고려해야 한다.
JobParameters는 업무 기준으로 설계해야 한다.
```

## 22. 3장 요약

```text
운영 배치는 실행보다 재실행과 장애 대응이 중요하다.
JobParameters는 업무 기준으로 잡아야 한다.
chunk size는 트랜잭션 크기다.
외부 API 호출은 DB rollback과 함께 취소되지 않는다.
인터페이스 테이블 패턴은 PLM, ERP, MES 연계에 적합하다.
Oracle에서는 row lock, undo, redo, 인덱스를 고려해야 한다.
실패 건수, skip 건수, write 건수를 반드시 로그로 남겨야 한다.
```
