# Spring Batch 02. Chunk 기반 코드 상세 설명

## 1. 예제 시나리오

이번 파일에서는 다음 흐름을 기준으로 설명한다.

```text
DB에서 회원 조회
→ 휴면 대상 회원만 필터링
→ 상태를 DORMANT로 변경
→ DB에 업데이트
```

구조:

```text
DormantMemberJob
 └─ dormantMemberStep
     ├─ ItemReader<Member>
     ├─ ItemProcessor<Member, Member>
     └─ ItemWriter<Member>
```

## 2. 예제 테이블

```sql
CREATE TABLE MEMBER (
    ID NUMBER PRIMARY KEY,
    NAME VARCHAR2(100),
    STATUS VARCHAR2(20),
    LAST_LOGIN_DATE DATE,
    UPDATED_AT DATE
);
```

예제 데이터:

```sql
INSERT INTO MEMBER (ID, NAME, STATUS, LAST_LOGIN_DATE, UPDATED_AT)
VALUES (1, 'Kim', 'ACTIVE', DATE '2025-01-01', SYSDATE);

INSERT INTO MEMBER (ID, NAME, STATUS, LAST_LOGIN_DATE, UPDATED_AT)
VALUES (2, 'Lee', 'ACTIVE', DATE '2026-04-01', SYSDATE);

INSERT INTO MEMBER (ID, NAME, STATUS, LAST_LOGIN_DATE, UPDATED_AT)
VALUES (3, 'Park', 'ACTIVE', DATE '2024-12-01', SYSDATE);
```

## 3. DTO 클래스

```java
@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
public class Member {

    private Long id;
    private String name;
    private String status;
    private LocalDate lastLoginDate;
    private LocalDateTime updatedAt;
}
```

설명:

```java
private Long id;
```

회원 PK다.

```java
private String status;
```

회원 상태다. 예제에서는 ACTIVE, DORMANT를 사용한다.

```java
private LocalDate lastLoginDate;
```

마지막 로그인 날짜다. 이 날짜를 기준으로 휴면 여부를 판단한다.

## 4. 전체 Job 설정 코드

```java
@Configuration
@RequiredArgsConstructor
public class DormantMemberJobConfig {

    private final DataSource dataSource;

    @Bean
    public Job dormantMemberJob(JobRepository jobRepository,
                                Step dormantMemberStep) {
        return new JobBuilder("dormantMemberJob", jobRepository)
                .start(dormantMemberStep)
                .build();
    }

    @Bean
    public Step dormantMemberStep(JobRepository jobRepository,
                                  PlatformTransactionManager transactionManager,
                                  ItemReader<Member> dormantMemberReader,
                                  ItemProcessor<Member, Member> dormantMemberProcessor,
                                  ItemWriter<Member> dormantMemberWriter) {
        return new StepBuilder("dormantMemberStep", jobRepository)
                .<Member, Member>chunk(100, transactionManager)
                .reader(dormantMemberReader)
                .processor(dormantMemberProcessor)
                .writer(dormantMemberWriter)
                .build();
    }

    @Bean
    public JdbcCursorItemReader<Member> dormantMemberReader() {
        return new JdbcCursorItemReaderBuilder<Member>()
                .name("dormantMemberReader")
                .dataSource(dataSource)
                .sql("SELECT ID, NAME, STATUS, LAST_LOGIN_DATE, UPDATED_AT " +
                     "FROM MEMBER " +
                     "WHERE STATUS = 'ACTIVE' " +
                     "ORDER BY ID")
                .rowMapper((rs, rowNum) -> {
                    Member member = new Member();
                    member.setId(rs.getLong("ID"));
                    member.setName(rs.getString("NAME"));
                    member.setStatus(rs.getString("STATUS"));
                    member.setLastLoginDate(rs.getDate("LAST_LOGIN_DATE").toLocalDate());

                    Timestamp updatedAt = rs.getTimestamp("UPDATED_AT");
                    if (updatedAt != null) {
                        member.setUpdatedAt(updatedAt.toLocalDateTime());
                    }

                    return member;
                })
                .build();
    }

    @Bean
    public ItemProcessor<Member, Member> dormantMemberProcessor() {
        return member -> {
            LocalDate dormantBaseDate = LocalDate.now().minusYears(1);

            if (member.getLastLoginDate().isAfter(dormantBaseDate)) {
                return null;
            }

            member.setStatus("DORMANT");
            member.setUpdatedAt(LocalDateTime.now());

            return member;
        };
    }

    @Bean
    public JdbcBatchItemWriter<Member> dormantMemberWriter() {
        return new JdbcBatchItemWriterBuilder<Member>()
                .dataSource(dataSource)
                .sql("UPDATE MEMBER " +
                     "SET STATUS = :status, UPDATED_AT = :updatedAt " +
                     "WHERE ID = :id")
                .beanMapped()
                .build();
    }
}
```

## 5. Job 코드 설명

```java
new JobBuilder("dormantMemberJob", jobRepository)
```

Job 이름은 dormantMemberJob이다. 이 이름은 Spring Batch 메타 테이블에 기록된다.

```java
.start(dormantMemberStep)
```

이 Job은 dormantMemberStep부터 실행된다.

Step이 여러 개라면 다음처럼 연결한다.

```java
return new JobBuilder("memberJob", jobRepository)
        .start(step1)
        .next(step2)
        .next(step3)
        .build();
```

## 6. Step 코드 설명

```java
.<Member, Member>chunk(100, transactionManager)
```

앞의 Member는 Reader가 읽는 타입이다.
뒤의 Member는 Writer가 쓰는 타입이다.

```text
ItemReader<Member>
ItemProcessor<Member, Member>
ItemWriter<Member>
```

즉, 읽는 데이터도 Member고 쓰는 데이터도 Member라는 뜻이다.

만약 CSV를 읽어서 Entity로 변환한다면 다음처럼 타입이 달라질 수 있다.

```java
.<MemberCsvRow, Member>chunk(100, transactionManager)
```

이 경우:

```text
Reader: MemberCsvRow 읽음
Processor: MemberCsvRow를 Member로 변환
Writer: Member 저장
```

## 7. chunk(100)의 의미

```java
chunk(100, transactionManager)
```

100건 단위로 트랜잭션을 커밋한다는 뜻이다.

흐름:

```text
1번 read/process
2번 read/process
...
100번 read/process
writer.write(100건)
commit
```

101번부터는 새로운 트랜잭션에서 처리된다.

## 8. Reader 코드 설명

```java
@Bean
public JdbcCursorItemReader<Member> dormantMemberReader() {
    return new JdbcCursorItemReaderBuilder<Member>()
            .name("dormantMemberReader")
            .dataSource(dataSource)
            .sql("SELECT ID, NAME, STATUS, LAST_LOGIN_DATE, UPDATED_AT " +
                 "FROM MEMBER " +
                 "WHERE STATUS = 'ACTIVE' " +
                 "ORDER BY ID")
            .rowMapper((rs, rowNum) -> {
                Member member = new Member();
                member.setId(rs.getLong("ID"));
                member.setName(rs.getString("NAME"));
                member.setStatus(rs.getString("STATUS"));
                member.setLastLoginDate(rs.getDate("LAST_LOGIN_DATE").toLocalDate());
                return member;
            })
            .build();
}
```

JdbcCursorItemReader는 DB Cursor 기반으로 데이터를 읽는 Reader다.

```text
SELECT 실행
→ ResultSet 열림
→ 한 행씩 read
→ Member 객체로 변환
```

```java
.name("dormantMemberReader")
```

Reader 이름이다. 재시작 상태 저장 등에 사용될 수 있으므로 명확하게 지정하는 것이 좋다.

```java
.dataSource(dataSource)
```

어떤 DB에서 읽을지 지정한다.

```java
.sql("SELECT ... ORDER BY ID")
```

읽을 데이터를 조회하는 SQL이다. ORDER BY를 넣는 이유는 재시작과 일관성 때문이다.

```java
.rowMapper((rs, rowNum) -> { ... })
```

DB 한 row를 Java 객체 하나로 바꾼다.

## 9. Processor 코드 설명

```java
@Bean
public ItemProcessor<Member, Member> dormantMemberProcessor() {
    return member -> {
        LocalDate dormantBaseDate = LocalDate.now().minusYears(1);

        if (member.getLastLoginDate().isAfter(dormantBaseDate)) {
            return null;
        }

        member.setStatus("DORMANT");
        member.setUpdatedAt(LocalDateTime.now());

        return member;
    };
}
```

Processor는 데이터를 가공하거나 필터링한다.

```java
if (member.getLastLoginDate().isAfter(dormantBaseDate)) {
    return null;
}
```

Processor에서 null을 반환하면 해당 데이터는 Writer로 넘어가지 않는다.
즉, 필터링된다.

예를 들어 Reader가 100건을 읽었는데 Processor가 30건을 null로 반환하면 Writer에는 70건만 넘어간다.

```text
readCount = 100
filterCount = 30
writeCount = 70
```

## 10. Writer 코드 설명

```java
@Bean
public JdbcBatchItemWriter<Member> dormantMemberWriter() {
    return new JdbcBatchItemWriterBuilder<Member>()
            .dataSource(dataSource)
            .sql("UPDATE MEMBER " +
                 "SET STATUS = :status, UPDATED_AT = :updatedAt " +
                 "WHERE ID = :id")
            .beanMapped()
            .build();
}
```

JdbcBatchItemWriter는 JDBC batch update를 사용해서 여러 건을 한 번에 처리하는 Writer다.
chunk size가 100이면 Writer는 최대 100건을 한 번에 받아서 update한다.

```java
.beanMapped()
```

Java Bean 필드명과 SQL 파라미터명을 자동 매핑한다.

```text
member.getStatus()    → :status
member.getUpdatedAt() → :updatedAt
member.getId()        → :id
```

## 11. Chunk 트랜잭션 흐름

chunk size가 3이라고 가정한다.

```text
트랜잭션 시작
read member1
process member1
read member2
process member2
read member3
process member3
writer.write([member1, member2, member3])
commit
```

writer에서 오류가 나면 다음처럼 된다.

```text
트랜잭션 시작
read 1
process 1
read 2
process 2
read 3
process 3
writer 실패
rollback
```

이 chunk 전체가 롤백된다.
단, 외부 API 호출 같은 DB 트랜잭션 밖의 작업은 롤백되지 않을 수 있다.

## 12. MyBatis Writer 예시

Spring Batch에서 Writer는 JdbcBatchItemWriter만 써야 하는 것이 아니다.
MyBatis Mapper를 호출하는 Custom Writer도 만들 수 있다.

### Mapper

```java
@Mapper
public interface MemberMapper {
    int updateDormantMember(Member member);
}
```

### XML

```xml
<update id="updateDormantMember">
    UPDATE MEMBER
    SET STATUS = #{status},
        UPDATED_AT = #{updatedAt}
    WHERE ID = #{id}
</update>
```

### Writer

```java
@Component
@RequiredArgsConstructor
public class MemberDormantWriter implements ItemWriter<Member> {

    private final MemberMapper memberMapper;

    @Override
    public void write(Chunk<? extends Member> chunk) {
        for (Member member : chunk) {
            memberMapper.updateDormantMember(member);
        }
    }
}
```

Spring Batch 5에서는 Writer 메서드가 List가 아니라 Chunk를 받는다.

```java
public void write(Chunk<? extends Member> chunk)
```

chunk 안의 데이터를 하나씩 꺼내 Mapper를 호출한다.

## 13. MyBatis Writer 주의점

위 방식은 이해하기 쉽지만 건별 update다.

```text
chunk 100건
→ updateDormantMember 100번 호출
```

성능이 중요하면 MyBatis foreach를 이용한 bulk update나 JDBC batch를 고려해야 한다.
다만 Oracle에서 대량 update는 undo, redo, 락 경합을 고려해야 한다.

## 14. CSV 파일 읽기 예시

CSV:

```csv
id,name,status,lastLoginDate
1,Kim,ACTIVE,2025-01-01
2,Lee,ACTIVE,2026-04-01
3,Park,ACTIVE,2024-12-01
```

DTO:

```java
@Getter
@Setter
public class MemberCsvRow {
    private Long id;
    private String name;
    private String status;
    private String lastLoginDate;
}
```

Reader:

```java
@Bean
public FlatFileItemReader<MemberCsvRow> memberCsvReader() {
    return new FlatFileItemReaderBuilder<MemberCsvRow>()
            .name("memberCsvReader")
            .resource(new ClassPathResource("member.csv"))
            .delimited()
            .names("id", "name", "status", "lastLoginDate")
            .targetType(MemberCsvRow.class)
            .linesToSkip(1)
            .build();
}
```

Processor:

```java
@Bean
public ItemProcessor<MemberCsvRow, Member> memberCsvProcessor() {
    return row -> {
        Member member = new Member();
        member.setId(row.getId());
        member.setName(row.getName());
        member.setStatus(row.getStatus());
        member.setLastLoginDate(LocalDate.parse(row.getLastLoginDate()));
        member.setUpdatedAt(LocalDateTime.now());
        return member;
    };
}
```

Step:

```java
@Bean
public Step csvToDbStep(JobRepository jobRepository,
                        PlatformTransactionManager transactionManager,
                        FlatFileItemReader<MemberCsvRow> memberCsvReader,
                        ItemProcessor<MemberCsvRow, Member> memberCsvProcessor,
                        JdbcBatchItemWriter<Member> memberWriter) {
    return new StepBuilder("csvToDbStep", jobRepository)
            .<MemberCsvRow, Member>chunk(100, transactionManager)
            .reader(memberCsvReader)
            .processor(memberCsvProcessor)
            .writer(memberWriter)
            .build();
}
```

## 15. JdbcPagingItemReader 예시

```java
@Bean
public JdbcPagingItemReader<Member> memberPagingReader() throws Exception {
    OraclePagingQueryProvider queryProvider = new OraclePagingQueryProvider();
    queryProvider.setSelectClause("SELECT ID, NAME, STATUS, LAST_LOGIN_DATE, UPDATED_AT");
    queryProvider.setFromClause("FROM MEMBER");
    queryProvider.setWhereClause("WHERE STATUS = 'ACTIVE'");

    Map<String, Order> sortKeys = new HashMap<>();
    sortKeys.put("ID", Order.ASCENDING);
    queryProvider.setSortKeys(sortKeys);

    return new JdbcPagingItemReaderBuilder<Member>()
            .name("memberPagingReader")
            .dataSource(dataSource)
            .queryProvider(queryProvider)
            .pageSize(100)
            .rowMapper((rs, rowNum) -> {
                Member member = new Member();
                member.setId(rs.getLong("ID"));
                member.setName(rs.getString("NAME"));
                member.setStatus(rs.getString("STATUS"));
                member.setLastLoginDate(rs.getDate("LAST_LOGIN_DATE").toLocalDate());
                return member;
            })
            .build();
}
```

## 16. Cursor Reader와 Paging Reader 차이

| 구분      | JdbcCursorItemReader      | JdbcPagingItemReader      |
| --------- | ------------------------- | ------------------------- |
| 방식      | 커서 기반                 | 페이지 쿼리 기반          |
| 대량 처리 | 가능                      | 가능                      |
| 정렬 기준 | SQL에 직접 작성           | sortKeys 필요             |
| 재시작    | 가능하지만 커서 특성 고려 | 정렬키 기반 재시작에 유리 |
| DB 부하   | 긴 커서 유지 가능         | 페이지 단위 조회          |

## 17. Skip 예시

```java
@Bean
public Step dormantMemberStep(JobRepository jobRepository,
                              PlatformTransactionManager transactionManager,
                              ItemReader<Member> reader,
                              ItemProcessor<Member, Member> processor,
                              ItemWriter<Member> writer) {
    return new StepBuilder("dormantMemberStep", jobRepository)
            .<Member, Member>chunk(100, transactionManager)
            .reader(reader)
            .processor(processor)
            .writer(writer)
            .faultTolerant()
            .skip(IllegalArgumentException.class)
            .skipLimit(10)
            .build();
}
```

```java
.faultTolerant()
```

장애 허용 설정을 시작한다.

```java
.skip(IllegalArgumentException.class)
```

IllegalArgumentException은 skip 대상으로 본다.

```java
.skipLimit(10)
```

최대 10건까지 skip을 허용한다.
11번째 오류가 발생하면 Step이 실패한다.

## 18. Retry 예시

```java
@Bean
public Step apiSendStep(JobRepository jobRepository,
                        PlatformTransactionManager transactionManager,
                        ItemReader<Member> reader,
                        ItemProcessor<Member, Member> processor,
                        ItemWriter<Member> writer) {
    return new StepBuilder("apiSendStep", jobRepository)
            .<Member, Member>chunk(50, transactionManager)
            .reader(reader)
            .processor(processor)
            .writer(writer)
            .faultTolerant()
            .retry(SocketTimeoutException.class)
            .retryLimit(3)
            .build();
}
```

외부 API가 멱등성을 보장해야 한다. 같은 요청이 2번 들어가도 결과가 꼬이지 않게 설계해야 한다.

## 19. 2장 요약

```text
Chunk 방식은 Reader, Processor, Writer로 나뉜다.
chunk size는 commit 단위다.
Processor에서 null을 반환하면 Writer로 넘어가지 않는다.
Writer는 chunk 단위로 호출된다.
JdbcBatchItemWriter는 대량 DB 저장에 적합하다.
MyBatis Writer도 가능하지만 성능을 고려해야 한다.
CSV Reader는 FlatFileItemReader를 사용한다.
대량 DB 조회에는 Cursor Reader 또는 Paging Reader를 사용한다.
Skip은 일부 오류 데이터를 넘기는 기능이다.
Retry는 일시적 장애를 재시도하는 기능이다.
```
