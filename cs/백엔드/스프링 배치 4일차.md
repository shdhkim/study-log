# Spring Batch 04. 인터페이스 테이블 설계 총정리

## 1. 인터페이스 테이블이란?

인터페이스 테이블은 서로 다른 시스템 사이에서 데이터를 직접 주고받지 않고, 중간 DB 테이블을 통해 안정적으로 데이터를 교환하기 위한 구조다.

예를 들어 다음과 같은 상황에서 많이 사용한다.

- 내부 시스템에서 외부 시스템으로 주문, 자재, 제품, 사용자, 기준정보를 전달
- 외부 시스템에서 생성된 결과 데이터를 내부 시스템으로 수신
- API 호출 실패 시 재처리 필요
- 대량 데이터를 일정 주기로 전달
- 송수신 이력, 오류 메시지, 처리 상태를 남겨야 하는 경우

즉, 인터페이스 테이블은 단순 임시 테이블이 아니라 시스템 연계의 안정성, 추적성, 재처리성을 보장하기 위한 운영 테이블이다.

---

## 2. 왜 바로 API 호출하지 않고 인터페이스 테이블을 쓰나?

### 2.1 직접 API 호출 방식

```text
업무 시스템 -> 외부 시스템 API
```

장점은 구조가 단순하다는 것이다.

하지만 다음 문제가 생길 수 있다.

- 외부 시스템 장애 시 업무 트랜잭션까지 실패할 수 있음
- 대량 데이터 처리에 부적합할 수 있음
- 실패 데이터 재처리가 어려움
- 어떤 데이터가 언제 전송됐는지 추적하기 어려움
- API 응답 지연이 업무 처리 속도에 영향을 줌

### 2.2 인터페이스 테이블 방식

```text
업무 시스템 -> 인터페이스 테이블 -> Spring Batch -> 외부 시스템
```

이 방식은 업무 처리와 외부 연계를 분리한다.

장점은 다음과 같다.

| 항목        | 설명                                                   |
| ----------- | ------------------------------------------------------ |
| 안정성      | 외부 시스템이 잠시 장애여도 원본 업무 처리는 유지 가능 |
| 재처리      | 실패 데이터만 다시 처리 가능                           |
| 추적성      | 송신/수신 이력, 상태, 오류 메시지를 DB에 남길 수 있음  |
| 대량 처리   | Chunk 단위로 안정적인 대량 처리 가능                   |
| 운영 편의성 | 상태값 기준으로 모니터링 가능                          |

---

## 3. 인터페이스 테이블의 기본 구조

인터페이스 테이블에는 보통 다음 컬럼들이 들어간다.

| 컬럼          | 의미                                |
| ------------- | ----------------------------------- |
| IF_ID         | 인터페이스 데이터 고유 ID           |
| BIZ_KEY       | 업무적으로 중복을 판단할 수 있는 키 |
| STATUS        | 처리 상태                           |
| REQUEST_BODY  | 송신 데이터 또는 수신 원문          |
| RESPONSE_BODY | 외부 시스템 응답                    |
| ERROR_MESSAGE | 오류 메시지                         |
| RETRY_COUNT   | 재시도 횟수                         |
| CREATED_AT    | 생성 시각                           |
| UPDATED_AT    | 수정 시각                           |
| PROCESSED_AT  | 처리 완료 시각                      |

---

## 4. 상태값 설계

인터페이스 테이블에서 가장 중요한 컬럼 중 하나가 상태값이다.

대표적인 상태값은 다음과 같다.

| 상태       | 의미            |
| ---------- | --------------- |
| READY      | 처리 대기       |
| PROCESSING | 처리 중         |
| SUCCESS    | 처리 성공       |
| FAIL       | 처리 실패       |
| HOLD       | 보류            |
| SKIP       | 의도적으로 제외 |

### 상태 흐름 예시

```text
READY -> PROCESSING -> SUCCESS
READY -> PROCESSING -> FAIL
FAIL  -> READY -> PROCESSING -> SUCCESS
```

실무에서는 `PROCESSING` 상태를 두는 것이 좋다. 여러 배치 서버가 동시에 실행될 때 같은 데이터를 중복 처리하는 문제를 줄일 수 있기 때문이다.

---

## 5. Header / Detail 구조

대량 연계나 마스터-상세 구조가 있는 데이터는 Header / Detail 테이블로 나누는 경우가 많다.

예를 들어 주문 데이터를 전송한다고 하면 다음처럼 나눌 수 있다.

```text
IF_ORDER_H
IF_ORDER_D
```

### Header 테이블

Header는 한 건의 인터페이스 요청 단위를 의미한다.

```text
주문번호 1개 = Header 1건
```

### Detail 테이블

Detail은 Header에 속한 상세 라인이다.

```text
주문번호 1개 안에 여러 품목 = Detail 여러 건
```

## 6. Oracle 기준 DDL 예제

### 6.1 Header 테이블

```sql
CREATE TABLE IF_ORDER_H (
    IF_ID           NUMBER          NOT NULL,
    ORDER_NO        VARCHAR2(50)    NOT NULL,
    CUSTOMER_CODE   VARCHAR2(50),
    STATUS          VARCHAR2(20)    DEFAULT 'READY' NOT NULL,
    REQUEST_BODY    CLOB,
    RESPONSE_BODY   CLOB,
    ERROR_MESSAGE   VARCHAR2(4000),
    RETRY_COUNT     NUMBER          DEFAULT 0 NOT NULL,
    CREATED_AT      DATE            DEFAULT SYSDATE NOT NULL,
    UPDATED_AT      DATE,
    PROCESSED_AT    DATE,
    CONSTRAINT PK_IF_ORDER_H PRIMARY KEY (IF_ID),
    CONSTRAINT UK_IF_ORDER_H_ORDER_NO UNIQUE (ORDER_NO)
);
```

### 6.2 Detail 테이블

```sql
CREATE TABLE IF_ORDER_D (
    IF_DETAIL_ID     NUMBER          NOT NULL,
    IF_ID            NUMBER          NOT NULL,
    LINE_NO          NUMBER          NOT NULL,
    ITEM_CODE        VARCHAR2(50)    NOT NULL,
    QTY              NUMBER          NOT NULL,
    PRICE            NUMBER,
    CREATED_AT       DATE            DEFAULT SYSDATE NOT NULL,
    CONSTRAINT PK_IF_ORDER_D PRIMARY KEY (IF_DETAIL_ID),
    CONSTRAINT FK_IF_ORDER_D_H FOREIGN KEY (IF_ID)
        REFERENCES IF_ORDER_H (IF_ID),
    CONSTRAINT UK_IF_ORDER_D_LINE UNIQUE (IF_ID, LINE_NO)
);
```

### 6.3 시퀀스

```sql
CREATE SEQUENCE SEQ_IF_ORDER_H
START WITH 1
INCREMENT BY 1
NOCACHE;

CREATE SEQUENCE SEQ_IF_ORDER_D
START WITH 1
INCREMENT BY 1
NOCACHE;
```

## 7. 인덱스 설계

배치에서는 보통 상태값과 생성일 기준으로 데이터를 조회한다.

```sql
CREATE INDEX IDX_IF_ORDER_H_01
ON IF_ORDER_H (STATUS, CREATED_AT);
```

실패건 재처리를 자주 한다면 다음 인덱스도 고려할 수 있다.

```sql
CREATE INDEX IDX_IF_ORDER_H_02
ON IF_ORDER_H (STATUS, RETRY_COUNT, UPDATED_AT);
```

업무 키 기준 조회가 많다면 다음 인덱스도 유용하다.

```sql
CREATE INDEX IDX_IF_ORDER_H_03
ON IF_ORDER_H (ORDER_NO);
```

단, 인덱스를 너무 많이 만들면 INSERT/UPDATE 성능이 떨어질 수 있다. 배치 조회 조건과 운영 조회 조건을 기준으로 최소한으로 설계하는 것이 좋다.

## 8. 중복 방지 설계

인터페이스 테이블에서 중복 방지는 매우 중요하다.

중복 방지를 하지 않으면 같은 주문, 같은 자재, 같은 기준정보가 여러 번 전송될 수 있다.

### 8.1 업무 키에 UNIQUE 제약조건 걸기

```sql
ALTER TABLE IF_ORDER_H
ADD CONSTRAINT UK_IF_ORDER_H_ORDER_NO UNIQUE (ORDER_NO);
```

이렇게 하면 같은 ORDER_NO가 두 번 들어가지 않는다.

### 8.2 MERGE 사용

이미 있으면 갱신하고, 없으면 삽입하는 방식이다.

```sql
MERGE INTO IF_ORDER_H T
USING (
    SELECT
        :orderNo AS ORDER_NO,
        :customerCode AS CUSTOMER_CODE
    FROM DUAL
) S
ON (T.ORDER_NO = S.ORDER_NO)
WHEN MATCHED THEN
    UPDATE SET
        T.CUSTOMER_CODE = S.CUSTOMER_CODE,
        T.UPDATED_AT = SYSDATE
WHEN NOT MATCHED THEN
    INSERT (
        IF_ID,
        ORDER_NO,
        CUSTOMER_CODE,
        STATUS,
        CREATED_AT
    ) VALUES (
        SEQ_IF_ORDER_H.NEXTVAL,
        S.ORDER_NO,
        S.CUSTOMER_CODE,
        'READY',
        SYSDATE
    );
```

### 8.3 MAX + 1 방식은 피하기

다음 방식은 동시성 문제가 생기기 쉽다.

```sql
SELECT MAX(IF_ID) + 1 FROM IF_ORDER_H;
```

동시에 두 트랜잭션이 실행되면 같은 값을 가져갈 수 있다. 따라서 Oracle에서는 시퀀스를 사용하는 것이 안전하다.

## 9. 처리 대상 조회 방식

배치에서 처리할 데이터는 보통 다음 조건으로 조회한다.

```sql
SELECT *
FROM IF_ORDER_H
WHERE STATUS = 'READY'
ORDER BY CREATED_AT, IF_ID;
```

하지만 여러 배치 인스턴스가 동시에 실행될 수 있다면 단순 조회만으로는 위험하다.

두 배치가 같은 READY 데이터를 동시에 읽을 수 있기 때문이다.

---

## 10. 중복 처리 방지를 위한 SELECT FOR UPDATE SKIP LOCKED

Oracle에서는 다음 패턴을 사용할 수 있다.

```sql
SELECT IF_ID
FROM IF_ORDER_H
WHERE STATUS = 'READY'
ORDER BY CREATED_AT, IF_ID
FOR UPDATE SKIP LOCKED;
```

`FOR UPDATE`는 선택한 행에 락을 건다.

`SKIP LOCKED`는 이미 다른 트랜잭션이 잡은 행은 건너뛰게 한다.

따라서 여러 배치 서버가 동시에 떠도 같은 데이터를 중복으로 잡을 가능성을 줄일 수 있다.

다만 Spring Batch의 Paging Reader와 `FOR UPDATE SKIP LOCKED`를 같이 쓸 때는 구현이 복잡해질 수 있다. 그래서 실무에서는 다음 방식도 많이 쓴다.

---

## 11. 먼저 PROCESSING으로 선점하는 방식

처리할 데이터를 먼저 `PROCESSING`으로 바꿔서 선점한다.

```sql
UPDATE IF_ORDER_H
SET STATUS = 'PROCESSING',
    UPDATED_AT = SYSDATE
WHERE IF_ID IN (
    SELECT IF_ID
    FROM (
        SELECT IF_ID
        FROM IF_ORDER_H
        WHERE STATUS = 'READY'
        ORDER BY CREATED_AT, IF_ID
    )
    WHERE ROWNUM <= 100
);
```

그 다음 `PROCESSING` 상태인 데이터를 읽어서 처리한다.

이 방식은 다음 장점이 있다.

- 처리 대상 선점이 명확함
- 중복 처리 가능성이 낮음
- 장애 발생 시 PROCESSING에 오래 머문 데이터를 복구 가능

단점도 있다.

- 선점 후 배치가 죽으면 PROCESSING 상태가 남을 수 있음
- 오래된 PROCESSING 데이터를 READY로 돌리는 복구 배치가 필요함

---

## 12. PROCESSING 복구 정책

배치가 중간에 죽으면 일부 데이터가 `PROCESSING` 상태로 남을 수 있다.

이를 방치하면 영원히 처리되지 않는다.

그래서 보통 다음 기준으로 복구한다.

```sql
UPDATE IF_ORDER_H
SET STATUS = 'READY',
    ERROR_MESSAGE = 'PROCESSING timeout recovery',
    UPDATED_AT = SYSDATE
WHERE STATUS = 'PROCESSING'
  AND UPDATED_AT < SYSDATE - (30 / 1440);
```

위 SQL은 30분 이상 PROCESSING 상태인 데이터를 READY로 되돌린다.

Oracle에서 날짜 계산은 다음과 같다.

```text
1일 = 1
1시간 = 1 / 24
1분 = 1 / 1440
30분 = 30 / 1440
```

## 13. 실패 데이터 관리

실패한 데이터는 `FAIL` 상태로 바꾸고 오류 메시지를 남긴다.

```sql
UPDATE IF_ORDER_H
SET STATUS = 'FAIL',
    ERROR_MESSAGE = :errorMessage,
    RETRY_COUNT = RETRY_COUNT + 1,
    UPDATED_AT = SYSDATE
WHERE IF_ID = :ifId;
```

오류 메시지는 너무 길 수 있으므로 Oracle `VARCHAR2(4000)`에 맞게 잘라서 저장하는 것이 좋다.

```java
private String cutErrorMessage(String message) {
    if (message == null) {
        return null;
    }
    return message.length() > 4000 ? message.substring(0, 4000) : message;
}
```

## 14. 재처리 설계

재처리는 보통 두 가지 방식이 있다.

### 14.1 수동 재처리

운영자가 실패 원인을 확인한 뒤 상태를 READY로 바꾼다.

```sql
UPDATE IF_ORDER_H
SET STATUS = 'READY',
    ERROR_MESSAGE = NULL,
    UPDATED_AT = SYSDATE
WHERE IF_ID = :ifId
  AND STATUS = 'FAIL';
```

### 14.2 자동 재처리

재시도 횟수가 일정 횟수 이하인 FAIL 데이터를 자동으로 다시 READY로 변경한다.

```sql
UPDATE IF_ORDER_H
SET STATUS = 'READY',
    UPDATED_AT = SYSDATE
WHERE STATUS = 'FAIL'
  AND RETRY_COUNT < 3;
```

단, 데이터 자체가 잘못된 경우 자동 재처리를 계속해도 계속 실패한다. 그래서 `RETRY_COUNT` 제한이 필요하다.

## 15. 송신 인터페이스와 수신 인터페이스

인터페이스 테이블은 방향에 따라 크게 두 종류로 나눌 수 있다.

| 구분            | 설명                                  |
| --------------- | ------------------------------------- |
| 송신 인터페이스 | 내부 DB 데이터를 외부 시스템으로 전달 |
| 수신 인터페이스 | 외부 시스템 데이터를 내부 DB로 반영   |

### 송신 인터페이스 흐름

```text
업무 테이블
  -> 인터페이스 테이블 생성
  -> Spring Batch가 READY 데이터 조회
  -> 외부 시스템 API 호출
  -> SUCCESS 또는 FAIL 업데이트
```

### 수신 인터페이스 흐름

```text
외부 시스템 API 호출
  -> 응답 데이터 수신
  -> 인터페이스 테이블 저장
  -> 검증
  -> 업무 테이블 반영
  -> SUCCESS 또는 FAIL 업데이트
```

## 16. 인터페이스 테이블과 업무 테이블 분리

인터페이스 테이블과 업무 테이블은 역할이 다르다.

| 구분        | 업무 테이블           | 인터페이스 테이블          |
| ----------- | --------------------- | -------------------------- |
| 목적        | 실제 업무 데이터 저장 | 연계 데이터 송수신 관리    |
| 상태        | 업무 상태             | 인터페이스 처리 상태       |
| 오류 메시지 | 보통 저장하지 않음    | 저장함                     |
| 재처리      | 업무 로직에 따라 다름 | 상태값 기반 재처리         |
| 삭제        | 신중함                | 보관 기간 후 아카이빙 가능 |

인터페이스 테이블을 업무 테이블처럼 쓰면 안 된다. 반대로 업무 테이블에 연계 오류 메시지나 재처리 상태를 과하게 넣는 것도 좋지 않다.

## 17. 보관 기간과 아카이빙

인터페이스 테이블은 시간이 지나면 데이터가 계속 쌓인다.

그래서 보관 정책이 필요하다.

예를 들어 다음과 같이 운영할 수 있다.

| 데이터         | 보관 기간                   |
| -------------- | --------------------------- |
| SUCCESS        | 3개월 또는 6개월            |
| FAIL           | 운영자가 확인할 때까지 보관 |
| 원문 전문 CLOB | 용량이 크면 별도 아카이빙   |

아카이빙 예시:

```sql
INSERT INTO IF_ORDER_H_HIST
SELECT *
FROM IF_ORDER_H
WHERE STATUS = 'SUCCESS'
  AND PROCESSED_AT < ADD_MONTHS(SYSDATE, -3);

DELETE FROM IF_ORDER_H
WHERE STATUS = 'SUCCESS'
  AND PROCESSED_AT < ADD_MONTHS(SYSDATE, -3);
```

대량 DELETE는 Undo/Redo 부담이 크므로 운영에서는 일정 건수씩 나누어 삭제하는 것이 안전하다.

## 18. 인터페이스 테이블 설계 체크리스트

| 체크 항목                        | 설명                             |
| -------------------------------- | -------------------------------- |
| 고유 ID가 있는가                 | IF_ID 같은 PK 필요               |
| 업무 중복 키가 있는가            | ORDER_NO, ITEM_CODE 등           |
| 상태값이 명확한가                | READY, PROCESSING, SUCCESS, FAIL |
| 오류 메시지를 저장하는가         | 장애 분석에 필요                 |
| 재시도 횟수를 관리하는가         | 무한 재시도 방지                 |
| 생성/수정/처리 시각이 있는가     | 운영 추적에 필요                 |
| 중복 방지 제약조건이 있는가      | UNIQUE 권장                      |
| 조회 조건에 맞는 인덱스가 있는가 | STATUS, CREATED_AT 등            |
| PROCESSING 복구 정책이 있는가    | 배치 중단 대비                   |
| 보관/삭제 정책이 있는가          | 테이블 비대화 방지               |

## 19. 핵심 정리

인터페이스 테이블 설계의 핵심은 단순 저장이 아니라 안정적인 연계 운영이다.

중요한 포인트는 다음과 같다.

1. 상태값 기반으로 처리 흐름을 관리한다.
2. 중복 방지를 위해 업무 키에 UNIQUE 제약조건을 둔다.
3. 실패 원인을 저장하고 실패건만 재처리할 수 있게 만든다.
4. 여러 배치가 동시에 돌아도 같은 데이터를 중복 처리하지 않도록 설계한다.
5. PROCESSING 상태가 오래 남는 경우를 복구할 수 있어야 한다.
6. SUCCESS 데이터가 무한히 쌓이지 않도록 아카이빙 정책을 둔다.
7. 업무 테이블과 인터페이스 테이블의 역할을 분리한다.
