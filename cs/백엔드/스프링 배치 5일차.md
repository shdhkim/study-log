# Spring Batch 05. 인터페이스 배치 코드 상세 정리

## 1. 목표

이 문서는 인터페이스 테이블을 Spring Batch로 처리하는 코드를 쉽게 설명한다.

다루는 흐름은 다음과 같다.

```text
인터페이스 테이블 READY 데이터 조회
  -> Processor에서 외부 시스템 호출 또는 데이터 검증
  -> Writer에서 SUCCESS / FAIL 상태 업데이트
```

예제는 다음 기준으로 작성한다.

- Spring Batch 5.x 스타일
- Java Config 방식
- Oracle DB 기준
- Chunk 기반 처리
- JdbcCursorItemReader / JdbcBatchItemWriter 사용
- 상태값 기반 재처리 구조

---

## 2. 예제 테이블

이전 장에서 만든 Header 테이블을 사용한다.

```sql
CREATE TABLE IF_ORDER_H (
    IF_ID           NUMBER          NOT NULL,
    ORDER_NO        VARCHAR2(50)    NOT NULL,
    CUSTOMER_CODE   VARCHAR2(50),
    STATUS          VARCHAR2(20)    DEFAULT 'READY' NOT NULL,
    REQUEST_BODY    CLOB,
    RESPONSE_BODY   CLOB,
    ERROR_MESSAGE   VARCHAR2(4000),
    RETRY_COUNT     NUMBER          DEFAULT 0 NOT NULL,
    CREATED_AT      DATE            DEFAULT SYSDATE NOT NULL,
    UPDATED_AT      DATE,
    PROCESSED_AT    DATE,
    CONSTRAINT PK_IF_ORDER_H PRIMARY KEY (IF_ID),
    CONSTRAINT UK_IF_ORDER_H_ORDER_NO UNIQUE (ORDER_NO)
);
```

## 3. 전체 구조

```text
InterfaceOrderJob
 └─ sendOrderStep
     ├─ ItemReader
     │   └─ READY 상태 데이터 조회
     ├─ ItemProcessor
     │   └─ 외부 시스템 호출 또는 데이터 변환
     └─ ItemWriter
         └─ 성공/실패 결과 업데이트
```

Spring Batch에서 Chunk 구조는 다음과 같다.

```text
read 1건
process 1건
read 1건
process 1건
...
chunk size만큼 모이면
write 여러 건
commit
```

예를 들어 chunk size가 100이면 100건을 읽고 처리한 뒤 한 번에 write하고 commit한다.

## 5. Reader 코드

Reader는 `READY` 상태 데이터를 조회한다.

```java
@Bean
public JdbcCursorItemReader<InterfaceOrder> interfaceOrderReader(DataSource dataSource) {
    return new JdbcCursorItemReaderBuilder<InterfaceOrder>()
            .name("interfaceOrderReader")
            .dataSource(dataSource)
            .sql("""
                    SELECT
                        IF_ID,
                        ORDER_NO,
                        CUSTOMER_CODE,
                        REQUEST_BODY,
                        RETRY_COUNT
                    FROM IF_ORDER_H
                    WHERE STATUS = 'READY'
                    ORDER BY CREATED_AT, IF_ID
                    """)
            .rowMapper((rs, rowNum) -> {
                InterfaceOrder order = new InterfaceOrder();
                order.setIfId(rs.getLong("IF_ID"));
                order.setOrderNo(rs.getString("ORDER_NO"));
                order.setCustomerCode(rs.getString("CUSTOMER_CODE"));
                order.setRequestBody(rs.getString("REQUEST_BODY"));
                order.setRetryCount(rs.getInt("RETRY_COUNT"));
                return order;
            })
            .build();
}
```

### 코드 설명

```java
.name("interfaceOrderReader")
```

Reader 이름이다. Spring Batch 메타 테이블에서 식별용으로 쓰일 수 있다.

```java
.dataSource(dataSource)
```

조회할 DB 연결 정보다.

```java
.sql(...)
```

실제로 실행할 조회 SQL이다.

```java
WHERE STATUS = 'READY'
```

아직 처리되지 않은 대기 데이터를 가져온다.

```java
.rowMapper(...)
```

ResultSet 한 행을 Java 객체로 바꾼다.

## 6. Processor 코드

Processor에서는 외부 시스템 호출, 데이터 검증, 변환을 수행한다.

여기서는 예제로 외부 API 호출을 흉내 낸다.

```java
@Component
public class InterfaceOrderProcessor implements ItemProcessor<InterfaceOrder, InterfaceOrderResult> {

    private final ExternalOrderClient externalOrderClient;

    public InterfaceOrderProcessor(ExternalOrderClient externalOrderClient) {
        this.externalOrderClient = externalOrderClient;
    }

    @Override
    public InterfaceOrderResult process(InterfaceOrder item) {
        try {
            validate(item);

            String response = externalOrderClient.sendOrder(item);

            return InterfaceOrderResult.success(item.getIfId(), response);
        } catch (Exception e) {
            return InterfaceOrderResult.fail(item.getIfId(), cutErrorMessage(e.getMessage()));
        }
    }

    private void validate(InterfaceOrder item) {
        if (item.getOrderNo() == null || item.getOrderNo().isBlank()) {
            throw new IllegalArgumentException("ORDER_NO is required");
        }

        if (item.getCustomerCode() == null || item.getCustomerCode().isBlank()) {
            throw new IllegalArgumentException("CUSTOMER_CODE is required");
        }
    }

    private String cutErrorMessage(String message) {
        if (message == null) {
            return null;
        }
        return message.length() > 4000 ? message.substring(0, 4000) : message;
    }
}
```

### 핵심 포인트

```java
public class InterfaceOrderProcessor implements ItemProcessor<InterfaceOrder, InterfaceOrderResult>
```

`InterfaceOrder`를 입력으로 받고, `InterfaceOrderResult`를 출력으로 반환한다.

```java
validate(item);
```

필수값 검증을 한다.

```java
String response = externalOrderClient.sendOrder(item);
```

외부 시스템으로 데이터를 보낸다.

```java
return InterfaceOrderResult.success(item.getIfId(), response);
```

성공 결과를 만든다.

```java
return InterfaceOrderResult.fail(item.getIfId(), cutErrorMessage(e.getMessage()));
```

예외가 발생해도 배치 전체를 죽이지 않고 해당 건만 실패 결과로 만든다.

## 7. 외부 시스템 Client 예제

실무에서는 `RestTemplate`, `WebClient`, 사내 HTTP Client 등을 사용할 수 있다.

```java
@Component
public class ExternalOrderClient {

    public String sendOrder(InterfaceOrder order) {
        // 실제 실무에서는 여기서 외부 API를 호출한다.
        // 예제에서는 성공 응답을 문자열로 반환한다.
        return "{\"result\":\"OK\",\"orderNo\":\"" + order.getOrderNo() + "\"}";
    }
}
```

실제 API 호출을 넣으면 다음과 같은 형태가 된다.

```java
@Component
public class ExternalOrderClient {

    private final RestTemplate restTemplate;

    public ExternalOrderClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    public String sendOrder(InterfaceOrder order) {
        String url = "https://external.example.com/api/orders";

        Map<String, Object> request = new HashMap<>();
        request.put("orderNo", order.getOrderNo());
        request.put("customerCode", order.getCustomerCode());
        request.put("body", order.getRequestBody());

        ResponseEntity<String> response = restTemplate.postForEntity(url, request, String.class);

        if (!response.getStatusCode().is2xxSuccessful()) {
            throw new IllegalStateException("External API failed. status=" + response.getStatusCode());
        }

        return response.getBody();
    }
}
```

## 8. Writer 코드

Writer는 처리 결과를 보고 SUCCESS 또는 FAIL로 업데이트한다.

```java
@Bean
public JdbcBatchItemWriter<InterfaceOrderResult> interfaceOrderWriter(DataSource dataSource) {
    return new JdbcBatchItemWriterBuilder<InterfaceOrderResult>()
            .dataSource(dataSource)
            .sql("""
                    UPDATE IF_ORDER_H
                    SET STATUS = CASE WHEN :success = 1 THEN 'SUCCESS' ELSE 'FAIL' END,
                        RESPONSE_BODY = :responseBody,
                        ERROR_MESSAGE = :errorMessage,
                        RETRY_COUNT = CASE WHEN :success = 1 THEN RETRY_COUNT ELSE RETRY_COUNT + 1 END,
                        UPDATED_AT = SYSDATE,
                        PROCESSED_AT = CASE WHEN :success = 1 THEN SYSDATE ELSE PROCESSED_AT END
                    WHERE IF_ID = :ifId
                    """)
            .itemSqlParameterSourceProvider(item -> {
                MapSqlParameterSource params = new MapSqlParameterSource();
                params.addValue("ifId", item.getIfId());
                params.addValue("success", item.isSuccess() ? 1 : 0);
                params.addValue("responseBody", item.getResponseBody());
                params.addValue("errorMessage", item.getErrorMessage());
                return params;
            })
            .build();
}
```

### 코드 설명

```sql
STATUS = CASE WHEN :success = 1 THEN 'SUCCESS' ELSE 'FAIL' END
```

성공이면 SUCCESS, 실패면 FAIL로 바꾼다.

```sql
RETRY_COUNT = CASE WHEN :success = 1 THEN RETRY_COUNT ELSE RETRY_COUNT + 1 END
```

실패한 경우에만 재시도 횟수를 증가시킨다.

```sql
PROCESSED_AT = CASE WHEN :success = 1 THEN SYSDATE ELSE PROCESSED_AT END
```

성공한 경우에만 처리 완료 시간을 기록한다.

## 9. Job / Step 설정 전체 코드

```java
@Configuration
public class InterfaceOrderBatchConfig {

    @Bean
    public Job interfaceOrderJob(
            JobRepository jobRepository,
            Step sendOrderStep
    ) {
        return new JobBuilder("interfaceOrderJob", jobRepository)
                .start(sendOrderStep)
                .build();
    }

    @Bean
    public Step sendOrderStep(
            JobRepository jobRepository,
            PlatformTransactionManager transactionManager,
            JdbcCursorItemReader<InterfaceOrder> interfaceOrderReader,
            InterfaceOrderProcessor interfaceOrderProcessor,
            JdbcBatchItemWriter<InterfaceOrderResult> interfaceOrderWriter
    ) {
        return new StepBuilder("sendOrderStep", jobRepository)
                .<InterfaceOrder, InterfaceOrderResult>chunk(100, transactionManager)
                .reader(interfaceOrderReader)
                .processor(interfaceOrderProcessor)
                .writer(interfaceOrderWriter)
                .build();
    }
}
```

### 코드 흐름

```java
new JobBuilder("interfaceOrderJob", jobRepository)
```

Job을 만든다.

```java
.start(sendOrderStep)
```

이 Job이 실행되면 `sendOrderStep`부터 실행한다.

```java
new StepBuilder("sendOrderStep", jobRepository)
```

Step을 만든다.

```java
.<InterfaceOrder, InterfaceOrderResult>chunk(100, transactionManager)
```

입력 타입은 `InterfaceOrder`, 출력 타입은 `InterfaceOrderResult`다.

Chunk 크기는 100이다.

```java
.reader(interfaceOrderReader)
.processor(interfaceOrderProcessor)
.writer(interfaceOrderWriter)
```

읽기, 처리, 쓰기 순서로 연결한다.

## 10. 이 코드의 문제점

위 코드는 이해하기 쉽지만 실무에서는 한 가지 문제가 있다.

Reader가 READY 데이터를 읽은 뒤, Writer에서 SUCCESS/FAIL로 바꾸기 전까지 상태가 READY로 남아 있다.

즉, 배치가 여러 개 동시에 실행되면 같은 READY 데이터를 중복으로 읽을 수 있다.

그래서 실무에서는 보통 다음 중 하나를 사용한다.

1. 배치를 단일 인스턴스로만 실행한다.
2. 실행 전에 READY 데이터를 PROCESSING으로 선점한다.
3. SELECT FOR UPDATE SKIP LOCKED를 사용한다.
4. 파티셔닝 기준으로 처리 범위를 분리한다.

## 11. PROCESSING 선점 Step 추가

실무적으로 가장 이해하기 쉬운 방법은 먼저 처리 대상을 PROCESSING으로 바꾸는 것이다.

전체 흐름은 다음과 같다.

```text
claimStep
  -> READY 일부를 PROCESSING으로 변경
sendOrderStep
  -> PROCESSING 데이터 처리
```

## 12. 선점 Tasklet 코드

```java
@Component
public class ClaimInterfaceOrderTasklet implements Tasklet {

    private final JdbcTemplate jdbcTemplate;

    public ClaimInterfaceOrderTasklet(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    @Override
    public RepeatStatus execute(StepContribution contribution, ChunkContext chunkContext) {
        int updated = jdbcTemplate.update("""
                UPDATE IF_ORDER_H
                SET STATUS = 'PROCESSING',
                    UPDATED_AT = SYSDATE
                WHERE IF_ID IN (
                    SELECT IF_ID
                    FROM (
                        SELECT IF_ID
                        FROM IF_ORDER_H
                        WHERE STATUS = 'READY'
                        ORDER BY CREATED_AT, IF_ID
                    )
                    WHERE ROWNUM <= 1000
                )
                """);

        contribution.incrementWriteCount(updated);

        return RepeatStatus.FINISHED;
    }
}
```

### 코드 설명

```java
implements Tasklet
```

Tasklet은 단일 작업을 수행할 때 사용한다.

```sql
WHERE STATUS = 'READY'
```

대기 상태 데이터만 잡는다.

```sql
WHERE ROWNUM <= 1000
```

한 번에 최대 1000건만 선점한다.

```sql
SET STATUS = 'PROCESSING'
```

이 배치가 처리할 대상으로 상태를 바꾼다.

## 14. 선점 Step + 처리 Step Job 구성

```java
@Configuration
public class InterfaceOrderBatchConfig {

    @Bean
    public Job interfaceOrderJob(
            JobRepository jobRepository,
            Step claimOrderStep,
            Step sendOrderStep
    ) {
        return new JobBuilder("interfaceOrderJob", jobRepository)
                .start(claimOrderStep)
                .next(sendOrderStep)
                .build();
    }

    @Bean
    public Step claimOrderStep(
            JobRepository jobRepository,
            PlatformTransactionManager transactionManager,
            ClaimInterfaceOrderTasklet tasklet
    ) {
        return new StepBuilder("claimOrderStep", jobRepository)
                .tasklet(tasklet, transactionManager)
                .build();
    }

    @Bean
    public Step sendOrderStep(

            JdbcCursorItemReader<InterfaceOrder> processingOrderReader,
            InterfaceOrderProcessor interfaceOrderProcessor,
            JdbcBatchItemWriter<InterfaceOrderResult> interfaceOrderWriter

                .reader(processingOrderReader)
                .processor(interfaceOrderProcessor)
                .writer(interfaceOrderWriter)
                .build();
    }
}
```

흐름은 다음과 같다.

```text

  -> READY 1000건을 PROCESSING으로 변경
```

## 15. 실패건 재처리 배치

실패 데이터 중 재시도 횟수가 3회 미만인 건만 READY로 되돌릴 수 있다.

```java
@Component
public class RetryFailedInterfaceOrderTasklet implements Tasklet {

    private final JdbcTemplate jdbcTemplate;

    public RetryFailedInterfaceOrderTasklet(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    @Override
    public RepeatStatus execute(StepContribution contribution, ChunkContext chunkContext) {
        int updated = jdbcTemplate.update("""
                UPDATE IF_ORDER_H
                SET STATUS = 'READY',
                    ERROR_MESSAGE = NULL,
                    UPDATED_AT = SYSDATE
                WHERE STATUS = 'FAIL'
                  AND RETRY_COUNT < 3
                """);

        contribution.incrementWriteCount(updated);

        return RepeatStatus.FINISHED;
    }
}
```

이 배치는 다음 역할을 한다.

```text
FAIL -> READY
```

단, `RETRY_COUNT < 3` 조건이 있으므로 무한 재처리를 막는다.

## 16. PROCESSING 복구 배치

배치가 중간에 죽으면 PROCESSING 상태가 남을 수 있다.

이를 복구하는 Tasklet이다.

```java
@Component
public class RecoverProcessingInterfaceOrderTasklet implements Tasklet {

    private final JdbcTemplate jdbcTemplate;

    public RecoverProcessingInterfaceOrderTasklet(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    @Override
    public RepeatStatus execute(StepContribution contribution, ChunkContext chunkContext) {
        int updated = jdbcTemplate.update("""
                UPDATE IF_ORDER_H
                SET STATUS = 'READY',
                    ERROR_MESSAGE = 'PROCESSING timeout recovery',
                    UPDATED_AT = SYSDATE
                WHERE STATUS = 'PROCESSING'
                  AND UPDATED_AT < SYSDATE - (30 / 1440)
                """);

        contribution.incrementWriteCount(updated);

        return RepeatStatus.FINISHED;
    }
}
```

이 배치는 다음 역할을 한다.

```text
오래된 PROCESSING -> READY
```

`30 / 1440`은 30분을 의미한다.

## 17. Skip / Retry 사용 시 주의점

Spring Batch에는 자체 Retry와 Skip 기능이 있다.

예를 들어 다음처럼 설정할 수 있다.

```java
@Bean
public Step sendOrderStep(
        JobRepository jobRepository,
        PlatformTransactionManager transactionManager,
        JdbcCursorItemReader<InterfaceOrder> reader,
        InterfaceOrderProcessor processor,
        JdbcBatchItemWriter<InterfaceOrderResult> writer
) {
    return new StepBuilder("sendOrderStep", jobRepository)
            .<InterfaceOrder, InterfaceOrderResult>chunk(100, transactionManager)
            .reader(reader)
            .processor(processor)
            .writer(writer)
            .faultTolerant()
            .retry(RuntimeException.class)
            .retryLimit(3)
            .skip(IllegalArgumentException.class)
            .skipLimit(100)
            .build();
}
```

하지만 인터페이스 배치에서는 주의해야 한다.

외부 API 호출이 이미 성공했는데, 배치 트랜잭션만 롤백되면 같은 데이터가 다시 전송될 수 있다.

즉, 외부 시스템 호출은 DB 트랜잭션으로 롤백되지 않는다.

```text
DB 업데이트 실패 -> 롤백 가능
외부 API 호출 성공 -> 롤백 불가능
```

그래서 외부 API 호출이 있는 배치에서는 멱등성 설계가 중요하다.

## 18. 멱등성 설계

멱등성이란 같은 요청을 여러 번 보내도 결과가 한 번 보낸 것과 같도록 만드는 것이다.

예를 들어 외부 시스템에 주문을 보낼 때 `ORDER_NO`를 고유 키로 사용하면 같은 주문이 두 번 생성되지 않게 할 수 있다.

```text
POST /orders
idempotencyKey = ORDER_NO
```

외부 시스템이 멱등성을 지원하지 않는다면 내부에서 중복 전송을 더 강하게 막아야 한다.

방법은 다음과 같다.

- 업무 키 기준 UNIQUE 제약조건 사용
- SUCCESS 데이터는 절대 재전송하지 않기
- 외부 요청 전문과 응답 전문 저장
- 외부 시스템 조회 API로 이미 처리됐는지 확인 후 전송
- API 요청에 고유 요청 ID 포함

## 19. 수신 인터페이스 배치 예시

수신 배치는 외부 시스템에서 데이터를 가져와 내부 DB에 반영한다.

흐름은 다음과 같다.

```text
외부 API 호출
  -> 응답 데이터 파싱
  -> 인터페이스 테이블 저장
  -> 업무 테이블 MERGE
  -> SUCCESS / FAIL 처리
```

단순 예제 Tasklet은 다음과 같다.

```java
@Component
public class ReceiveOrderTasklet implements Tasklet {

    private final ExternalOrderReceiveClient client;
    private final JdbcTemplate jdbcTemplate;

    public ReceiveOrderTasklet(ExternalOrderReceiveClient client, JdbcTemplate jdbcTemplate) {
        this.client = client;
        this.jdbcTemplate = jdbcTemplate;
    }

    @Override
    public RepeatStatus execute(StepContribution contribution, ChunkContext chunkContext) {
        List<ReceivedOrder> orders = client.getOrders();

        for (ReceivedOrder order : orders) {
            jdbcTemplate.update("""
                    MERGE INTO IF_RECEIVE_ORDER T
                    USING (
                        SELECT ? AS ORDER_NO,
                               ? AS CUSTOMER_CODE,
                               ? AS RAW_BODY
                        FROM DUAL
                    ) S
                    ON (T.ORDER_NO = S.ORDER_NO)
                    WHEN MATCHED THEN
                        UPDATE SET
                            T.CUSTOMER_CODE = S.CUSTOMER_CODE,
                            T.RAW_BODY = S.RAW_BODY,
                            T.STATUS = 'READY',
                            T.UPDATED_AT = SYSDATE
                    WHEN NOT MATCHED THEN
                        INSERT (
                            IF_ID,
                            ORDER_NO,
                            CUSTOMER_CODE,
                            RAW_BODY,
                            STATUS,
                            CREATED_AT
                        ) VALUES (
                            SEQ_IF_RECEIVE_ORDER.NEXTVAL,
                            S.ORDER_NO,
                            S.CUSTOMER_CODE,
                            S.RAW_BODY,
                            'READY',
                            SYSDATE
                        )
                    """,
                    order.getOrderNo(),
                    order.getCustomerCode(),
                    order.getRawBody()
            );
        }

        contribution.incrementWriteCount(orders.size());

        return RepeatStatus.FINISHED;
    }
}
```

수신 배치에서는 `MERGE`를 많이 사용한다. 같은 데이터를 중복 수신해도 한 번만 저장하거나 갱신하기 위해서다.

## 20. 스케줄러 연동

Spring Scheduler로 배치를 실행할 수 있다.

```java
@Component
public class InterfaceBatchScheduler {

    private final JobLauncher jobLauncher;
    private final Job interfaceOrderJob;

    public InterfaceBatchScheduler(JobLauncher jobLauncher, Job interfaceOrderJob) {
        this.jobLauncher = jobLauncher;
        this.interfaceOrderJob = interfaceOrderJob;
    }

    @Scheduled(cron = "0 */5 * * * *")
    public void runInterfaceOrderJob() throws Exception {
        JobParameters jobParameters = new JobParametersBuilder()
                .addLong("timestamp", System.currentTimeMillis())
                .toJobParameters();

        jobLauncher.run(interfaceOrderJob, jobParameters);
    }
}
```

### 중요한 점

```java
.addLong("timestamp", System.currentTimeMillis())
```

Spring Batch는 같은 Job 이름과 같은 Job Parameter로 실행된 Job을 같은 Job Instance로 본다.

그래서 매번 새로 실행하려면 파라미터를 다르게 넣어야 한다.

## 21. 운영 로그 Listener

Job 실행 결과를 로그로 남기려면 Listener를 사용할 수 있다.

```java
@Component
public class InterfaceJobListener implements JobExecutionListener {

    @Override
    public void beforeJob(JobExecution jobExecution) {
        System.out.println("Job started: " + jobExecution.getJobInstance().getJobName());
    }

    @Override
    public void afterJob(JobExecution jobExecution) {
        System.out.println("Job ended: " + jobExecution.getJobInstance().getJobName());
        System.out.println("Status: " + jobExecution.getStatus());
        System.out.println("ExitStatus: " + jobExecution.getExitStatus());
    }
}
```

Job에 연결한다.

```java
@Bean
public Job interfaceOrderJob(
        JobRepository jobRepository,
        Step claimOrderStep,
        Step sendOrderStep,
        InterfaceJobListener listener
) {
    return new JobBuilder("interfaceOrderJob", jobRepository)
            .listener(listener)
            .start(claimOrderStep)
            .next(sendOrderStep)
            .build();
}
```

## 22. 실무에서 자주 나는 문제

### 22.1 같은 데이터가 두 번 전송됨

원인:

- READY 데이터를 여러 배치가 동시에 읽음
- 외부 API 성공 후 DB 업데이트 실패
- 재처리 조건이 잘못됨

대응:

- PROCESSING 선점 방식 사용
- 업무 키 UNIQUE 제약조건 사용
- 외부 시스템 멱등성 키 사용
- SUCCESS 데이터는 재처리 대상에서 제외

---

### 22.2 PROCESSING 상태가 계속 남음

원인:

- 배치 서버가 중간에 종료됨
- 외부 API 응답 지연으로 타임아웃 발생
- Writer 실행 전에 장애 발생

대응:

- PROCESSING timeout 복구 배치 운영
- UPDATED_AT 기준으로 오래된 건 READY 복구
- 복구 이력을 ERROR_MESSAGE에 남김

---

### 22.3 FAIL이 너무 많이 쌓임

원인:

- 데이터 자체 오류
- 외부 시스템 장애
- 필수값 누락
- API 규격 변경

대응:

- ERROR_MESSAGE 저장
- 실패 사유별 집계 쿼리 작성
- 재처리 횟수 제한
- 운영자가 직접 수정 후 재처리할 수 있는 화면 제공

---

### 22.4 배치가 너무 느림

원인:

- STATUS 조건 인덱스 없음
- 너무 작은 chunk size
- 외부 API를 한 건씩 동기 호출
- CLOB 대량 조회
- UPDATE 대상이 많음

대응:

- `(STATUS, CREATED_AT)` 인덱스 생성
- chunk size 조정
- 외부 API bulk 호출 가능 여부 검토
- 필요한 컬럼만 조회
- 오래된 SUCCESS 데이터 아카이빙

---

## 23. 실무 추천 구조

송신 인터페이스 배치는 다음 구조를 추천한다.

```text
1. recoverProcessingStep
   - 오래된 PROCESSING을 READY로 복구

2. retryFailStep
   - 재시도 가능한 FAIL을 READY로 변경

3. claimStep
   - READY 일부를 PROCESSING으로 선점

4. sendStep
   - PROCESSING 데이터를 외부 시스템으로 전송
   - SUCCESS / FAIL 업데이트
```

Job 구성 예시는 다음과 같다.

```java
@Bean
public Job interfaceOrderJob(
        JobRepository jobRepository,
        Step recoverProcessingStep,
        Step retryFailStep,
        Step claimOrderStep,
        Step sendOrderStep
) {
    return new JobBuilder("interfaceOrderJob", jobRepository)
            .start(recoverProcessingStep)
            .next(retryFailStep)
            .next(claimOrderStep)
            .next(sendOrderStep)
            .build();
}
```

이 구조의 장점은 다음과 같다.

- 중간 장애 복구 가능
- 실패건 자동 재처리 가능
- 처리 대상 선점 가능
- 중복 전송 가능성 감소
- 운영자가 상태값만 보고 흐름을 이해하기 쉬움

---

## 24. 핵심 정리

인터페이스 배치 코드는 단순히 데이터를 읽고 쓰는 코드가 아니다.

실무에서는 다음이 중요하다.

1. READY, PROCESSING, SUCCESS, FAIL 상태 흐름을 명확히 만든다.
2. 여러 배치가 동시에 떠도 같은 데이터를 중복 처리하지 않게 한다.
3. 실패한 데이터는 오류 메시지와 재시도 횟수를 남긴다.
4. PROCESSING 상태가 오래 남는 경우 복구 배치를 둔다.
5. 외부 API 호출은 DB 트랜잭션으로 롤백되지 않으므로 멱등성 설계가 필요하다.
6. SUCCESS 데이터 재전송은 최대한 막아야 한다.
7. Chunk size, 인덱스, API 호출 방식이 성능에 큰 영향을 준다.
8. 운영자는 상태값, 오류 메시지, 처리 시각만 보고도 문제를 추적할 수 있어야 한다.
