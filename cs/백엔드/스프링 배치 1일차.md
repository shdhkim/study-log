# Spring Batch 01. 핵심 구조와 실행 흐름

## 1. Spring Batch가 필요한 이유

Spring Batch는 대량 데이터를 안정적으로 처리하기 위한 배치 프레임워크다.
웹 요청은 사용자의 즉시 응답이 목적이지만, 배치는 정해진 시간이나 수동 실행으로 대량 데이터를 처리하고 실행 이력을 남기는 것이 목적이다.

예시:

```text
매일 새벽 주문 정산
PLM 데이터를 ERP 인터페이스 테이블로 전송
대량 CSV 업로드 처리
월말 재고 마감
오래된 로그 삭제
외부 시스템 데이터 동기화
```

## 2. 핵심 용어

| 개념             | 의미                                      |
| ---------------- | ----------------------------------------- |
| Job              | 하나의 배치 작업 전체                     |
| Step             | Job 안의 개별 처리 단계                   |
| Tasklet          | 단순한 1회성 작업 방식                    |
| Chunk            | 데이터를 일정 단위로 끊어서 처리하는 방식 |
| ItemReader       | 데이터를 읽는 역할                        |
| ItemProcessor    | 데이터를 검증, 변환, 필터링하는 역할      |
| ItemWriter       | 데이터를 저장하거나 전송하는 역할         |
| JobRepository    | 배치 실행 이력을 저장하는 저장소          |
| JobLauncher      | Job을 실행하는 객체                       |
| JobParameters    | Job 실행 시 전달하는 파라미터             |
| JobExecution     | Job 실행 1회에 대한 실행 정보             |
| StepExecution    | Step 실행 1회에 대한 실행 정보            |
| ExecutionContext | 재시작을 위해 상태를 저장하는 공간        |

## 3. Job과 Step 관계

```text
Job: dailyProductSyncJob

  Step 1: PLM에서 대상 제품 조회
  Step 2: ERP 전송용 데이터 변환
  Step 3: ERP 인터페이스 테이블에 저장
  Step 4: 결과 로그 저장
```

Spring Batch 구조:

```text
Job
 ├─ Step
 │   ├─ ItemReader
 │   ├─ ItemProcessor
 │   └─ ItemWriter
 └─ Step
     └─ Tasklet
```

## 4. Chunk 방식

Chunk 방식은 Spring Batch에서 가장 많이 쓰는 방식이다.
공식 문서 기준으로 Spring Batch의 chunk-oriented processing은 데이터를 하나씩 읽고, 일정 개수의 chunk를 만든 뒤, 트랜잭션 경계 안에서 write하고 commit하는 방식이다.

```text
read 1건
process 1건
read 1건
process 1건
...
commit interval만큼 모이면
writer가 한 번에 저장
commit
```

chunk size가 3이면 다음처럼 동작한다.

```text
1번 데이터 read
1번 데이터 process
2번 데이터 read
2번 데이터 process
3번 데이터 read
3번 데이터 process
writer.write([1번, 2번, 3번])
commit
```

중요한 점은 writer가 매 row마다 호출되는 것이 아니라 chunk 단위로 호출된다는 것이다.

## 5. Tasklet 방식과 Chunk 방식 차이

| 구분        | Tasklet                             | Chunk                      |
| ----------- | ----------------------------------- | -------------------------- |
| 목적        | 단순 작업                           | 대량 데이터 처리           |
| 구조        | execute 메서드 하나                 | Reader, Processor, Writer  |
| 예시        | 파일 삭제, 상태 변경, 프로시저 호출 | CSV 읽기, DB 대량 이관     |
| 트랜잭션    | Step 단위 단순 처리                 | chunk 단위 커밋            |
| 재시작 처리 | 직접 관리가 많음                    | Reader 상태 저장 지원 가능 |

## 6. Tasklet 예시

```java
@Bean
public Step deleteOldFileStep(JobRepository jobRepository,
                              PlatformTransactionManager transactionManager) {
    return new StepBuilder("deleteOldFileStep", jobRepository)
            .tasklet((contribution, chunkContext) -> {
                System.out.println("오래된 파일 삭제");
                return RepeatStatus.FINISHED;
            }, transactionManager)
            .build();
}
```

### 코드 설명

```java
new StepBuilder("deleteOldFileStep", jobRepository)
```

Step 이름과 JobRepository를 지정한다.

```java
.tasklet(...)
```

이 Step은 Chunk 방식이 아니라 Tasklet 방식으로 실행된다는 뜻이다.

```java
RepeatStatus.FINISHED
```

Tasklet 작업이 끝났다는 의미다.

## 7. Spring Batch 5.x 기본 의존성

Spring Boot 3.x, Spring Batch 5.x 기준 예시다.

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-batch'
    implementation 'org.springframework.boot:spring-boot-starter-jdbc'
    runtimeOnly 'com.h2database:h2'
}
```

Oracle 사용 시:

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-batch'
    implementation 'org.springframework.boot:spring-boot-starter-jdbc'
    runtimeOnly 'com.oracle.database.jdbc:ojdbc11'
}
```

Spring Batch 5는 Spring Framework 6 기반이고 Java 17 이상이 필요하다.

## 8. application.yml 기본 설정

H2 예시:

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:batchdb
    driver-class-name: org.h2.Driver
    username: sa
    password:

  batch:
    jdbc:
      initialize-schema: always
    job:
      enabled: false
```

설명:

```yaml
spring.batch.jdbc.initialize-schema: always
```

Spring Batch 메타 테이블을 자동 생성한다.

```yaml
spring.batch.job.enabled: false
```

애플리케이션 실행 시 Job이 자동으로 실행되지 않게 한다.
운영에서는 특정 Job만 원하는 시점에 실행하기 위해 false로 두는 경우가 많다.

Oracle 예시:

```yaml
spring:
  datasource:
    url: jdbc:oracle:thin:@localhost:1521/XEPDB1
    username: BATCH_USER
    password: 1234
    driver-class-name: oracle.jdbc.OracleDriver

  batch:
    jdbc:
      initialize-schema: never
    job:
      enabled: false
```

운영 DB에서는 보통 Batch 메타 테이블을 자동 생성하지 않고 스크립트로 직접 생성한다.

## 9. Spring Batch 메타 테이블

| 테이블                       | 의미                   |
| ---------------------------- | ---------------------- |
| BATCH_JOB_INSTANCE           | Job의 논리적 실행 단위 |
| BATCH_JOB_EXECUTION          | Job의 실제 실행 이력   |
| BATCH_JOB_EXECUTION_PARAMS   | JobParameters 저장     |
| BATCH_STEP_EXECUTION         | Step 실행 이력         |
| BATCH_STEP_EXECUTION_CONTEXT | Step 재시작 상태 저장  |
| BATCH_JOB_EXECUTION_CONTEXT  | Job 재시작 상태 저장   |

같은 Job이라도 파라미터가 다르면 다른 JobInstance가 된다.

```text
dailyJob, date=2026-04-26
dailyJob, date=2026-04-27
```

둘은 서로 다른 JobInstance다.

## 10. 가장 기본적인 Job 코드

```java
@Configuration
public class SimpleJobConfig {

    @Bean
    public Job simpleJob(JobRepository jobRepository,
                         Step simpleStep) {
        return new JobBuilder("simpleJob", jobRepository)
                .start(simpleStep)
                .build();
    }

    @Bean
    public Step simpleStep(JobRepository jobRepository,
                           PlatformTransactionManager transactionManager) {
        return new StepBuilder("simpleStep", jobRepository)
                .tasklet((contribution, chunkContext) -> {
                    System.out.println("Simple Step 실행");
                    return RepeatStatus.FINISHED;
                }, transactionManager)
                .build();
    }
}
```

### 코드 설명

```java
@Configuration
```

Spring 설정 클래스라는 뜻이다.

```java
@Bean
public Job simpleJob(...)
```

Spring Bean으로 Job을 등록한다.

```java
new JobBuilder("simpleJob", jobRepository)
```

Job 이름을 simpleJob으로 지정한다. JobRepository는 실행 이력을 저장하는 저장소다.

```java
.start(simpleStep)
```

이 Job은 simpleStep부터 시작한다.

```java
.tasklet((contribution, chunkContext) -> { ... }, transactionManager)
```

Tasklet 방식 Step을 만든다. 트랜잭션 매니저도 함께 지정한다.

## 11. JobLauncher로 직접 실행하기

```java
@RestController
@RequiredArgsConstructor
public class BatchController {

    private final JobLauncher jobLauncher;
    private final Job simpleJob;

    @PostMapping("/batch/simple")
    public String runSimpleJob() throws Exception {
        JobParameters jobParameters = new JobParametersBuilder()
                .addLong("requestTime", System.currentTimeMillis())
                .toJobParameters();

        jobLauncher.run(simpleJob, jobParameters);

        return "배치 실행 요청 완료";
    }
}
```

### 코드 설명

```java
private final JobLauncher jobLauncher;
```

Job을 실행하는 객체다.

```java
private final Job simpleJob;
```

실행할 Job Bean을 주입받는다.

```java
new JobParametersBuilder()
```

Job 실행 파라미터를 만든다.

```java
.addLong("requestTime", System.currentTimeMillis())
```

매번 다른 JobInstance로 실행하기 위해 현재 시간을 파라미터로 넣는다.

```java
jobLauncher.run(simpleJob, jobParameters);
```

실제 Job을 실행한다.

## 12. JobParameters가 중요한 이유

Spring Batch는 Job 이름과 JobParameters 조합으로 JobInstance를 구분한다.

```text
JobInstance = Job 이름 + JobParameters
```

이미 성공한 JobInstance를 같은 파라미터로 다시 실행하면 `JobInstanceAlreadyCompleteException`이 날 수 있다.

테스트할 때는 보통 유니크한 파라미터를 넣는다.

```java
JobParameters jobParameters = new JobParametersBuilder()
        .addLong("time", System.currentTimeMillis())
        .toJobParameters();
```

하지만 운영에서는 무조건 time만 넣으면 재시작 관리가 어려워진다.
운영에서는 보통 업무 기준 파라미터를 명확히 넣는다.

```text
baseDate=2026-04-26
plantCode=M01
interfaceType=PRODUCT
```

## 13. 실행 상태

| 상태      | 의미                 |
| --------- | -------------------- |
| STARTING  | 시작 중              |
| STARTED   | 실행 중              |
| COMPLETED | 정상 완료            |
| FAILED    | 실패                 |
| STOPPING  | 중지 중              |
| STOPPED   | 중지됨               |
| ABANDONED | 재시작 대상에서 제외 |

StepExecution에는 readCount, writeCount, filterCount, commitCount, rollbackCount, skipCount 등이 기록된다.

## 14. 기본 실행 흐름

```text
1. JobLauncher가 Job 실행 요청
2. JobRepository에 JobExecution 생성
3. Job 안의 Step 실행
4. StepExecution 생성
5. Reader, Processor, Writer 실행
6. chunk 단위로 commit
7. StepExecution 상태 업데이트
8. JobExecution 상태 업데이트
9. 성공 또는 실패 기록
```

## 15. 1장 요약

```text
Job은 배치 작업 전체다.
Step은 Job 안의 처리 단계다.
Tasklet은 단순 작업에 적합하다.
Chunk는 대량 데이터 처리에 적합하다.
Reader는 읽고, Processor는 가공하고, Writer는 쓴다.
JobRepository는 실행 이력을 저장한다.
JobParameters는 JobInstance를 구분한다.
운영에서는 업무 기준 파라미터를 명확히 설계해야 한다.
```
